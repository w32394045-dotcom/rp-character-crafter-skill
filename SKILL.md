## ⚡ 快速开始（TL;DR — 30 秒学会用本 Skill）

你只需要记住这三条路径：

### 路径 A：创作新角色
```
用户说「写个角色卡」→ Step 1 需求收集（COT穷举追问至少3个缺口）
  → Step 1.5 确认需求 → Step 2 角色内核（5问：想要/最怕/底线/矛盾/言行不一）
  → Step 3 分层构建（表层/中层/深层）→ Step 4 格式输出（选格式出卡）
  → Step 5 反馈迭代（改最多3轮）
```

### 路径 B：从对话反推角色（蒸馏）
```
用户有对话记录 → 模式A（直接分析对话）
  → D1 输入检查 + D1.5 偏见补偿 → D2 五维分析 + D3 三重交叉验证
  → D4 缺口识别 → D5 输出角色卡+蒸馏报告

用户只有首句 → 模式B（主动采样）
  → S1 制定采样计划（用户确认）→ S2 逐轮采样 → S3 质量检查
  → S4 汇总 → 复用 D2-D5 流程
```

### 路径 C：从字幕文件提取角色（字幕蒸馏）
```
用户有 .ass/.srt/.ssa 字幕 → 模式C（字幕蒸馏）
  → C1 字幕解析与角色识别（自动拆分+统计）
  → C2 字幕专属六维分析（台词风格/关系网络/说话方式/行为推断/视觉线索/角色弧线）
  → C3 置信度体系（含「编剧明示」和「样式推断」层级）
  → C4 CHECKPOINT → C5 输出（单角色/多角色批量/角色卡+字幕世界书）
  → C6 可选：自动提取字幕世界书
```

### 核心纪律（不可违反）
1. **不知道就问** — 用户没说全=必须追问，至少3个问题一次问
2. **出卡前 CHECKPOINT** — Step 1/2/3/4/5 每个阶段结束必须确认
3. **回复格式强制嵌入** — 括号动作+裸对话+1-3段，写入 system_prompt/mes_example
4. **蒸馏不编造** — 每个特征标置信度，没出现的信息标注「推测/不足」
5. **字幕蒸馏区分来源** — 区分「编剧台词」vs「即兴对话」，标注翻译腔风险，ASS样式线索仅作参考

> 熟练后直接看工作流（§2）和格式指南（§3）的详细规格。

## 核心理念

优秀的角色卡 = **可被 AI 理解并一致演绎的人格系统**，而非一堆属性的罗列。

好的角色卡让 AI 知道：
- **我是谁** —— 身份、背景、世界观中的位置
- **我怎么想** —— 价值观、动机、信念体系
- **我怎么说话** —— 语气、用词、句式节奏
- **我怎么反应** —— 面对不同情境的行为模式
- **我有什么不能碰** —— 禁忌、创伤、性格盲区

### ⚠️ 抗幻觉守则 + 主动询问规则（所有操作必须遵守）

> **宁缺毋滥。不知道就问，用户没说的就要。除非用户明确说「不用问」，否则默认必须追问到底。**

#### 主动询问规则（新增，最高优先级）

| 场景 | 必须行为 |
|------|---------|
| 用户只说了一个角色概念（如「写个病娇」） | **必须追问**缺失的关键信息：至少问世界观、与用户关系、输出格式。不得直接开写 |
| 用户给了部分信息但有明显缺口（有性格没外貌） | **必须追问**缺口信息：「角色外貌/背景故事/……还没提供，要补充吗？」|
| 用户说「你看着办」| 先问发挥边界，见下方例外规则 |
| 用户说「不用问」「你定就行」| **例外生效**，可以自由创作 |
| 蒸馏时对话数据不足 | **必须告诉用户**数据量问题，询问是否需要补充或接受当前质量 |
| 用户要求模式B采样 | **必须展示采样计划给用户确认**后才能开始 |
| 用户提供字幕文件（模式C）| **必须展示角色列表给用户确认**后再选择目标角色分析 |
| 字幕文件角色标注缺失/不完整 | **必须追问**角色名或样例台词，不得仅靠聚类猜测 |
| 出卡后 | **必须执行 Step 5 反馈迭代**，问用户要不要改 |

**核心原则**：agent 的默认状态是**主动要信息**，不是等投喂。用户沉默或信息不全 = agent 继续追问，**直到用户主动说出「不用问」为止**。

#### 例外规则（以下短语触发后可自由创作，无需逐个确认）

- 「全部由你定」/「你自由发挥」/「你看着写」
- 「在这个世界观里随便写」
- 「按典型设定写就行」
- 「不用问」/「你定就行」/「别问了直接写」

**即使触发例外**，也需在 `creator_notes` 标注「部分内容由 AI 在用户授权下补充」。

## 工作流

### Step 1: 需求收集（强制COT全面追问）

> **禁止偷懒。禁止只问一两个问题就开写。必须用 COT（Chain of Thought）穷举所有缺失维度后再问。**

#### 1.1 内部COT分析（先思考，再开口，不输出给用户）

收到用户请求后，**先在内部执行完整的思维链分析**，按以下模板穷举所有维度：

```markdown
【COT 需求分析 - 内部使用，不输出】

一、用户已提供的信息
- 角色概念：[已提供/未提供]
- 世界观：[已提供/未提供]
- 外貌：[已提供/未提供]
- 性格：[已提供/未提供]
- 背景故事：[已提供/未提供]
- 与用户的关系：[已提供/未提供]
- 语言风格：[已提供/未提供]
- 输出格式：[已提供/未提供]
- NSFW/特殊要求：[已提供/未提供]

二、缺失维度（需要问的）
列出所有「未提供」或「只提供了一半」的维度，至少 3 个。
一个都不缺 → 确认并进入 Step 2。
有缺口 → 进入第三部分。

三、综合思考
- 基于已有信息，我初步认为这个角色是 [类型]，ta 的核心矛盾可能是 [推测]
- 需要重点确认的是 [1-2个关键问题]
- 用户可能的深层需求是 [推测]
```

**COT 执行约束**：
- 至少列出 3 个缺失维度才允许追问。如果用户确实给了所有信息（极少见），直接确认进入 Step 2
- COT 中「综合思考」部分必须给出**推测性总结**——不仅要发现缺口，还要给出基于已有信息的初步判断
- **禁止**只问一个「还有什么要补充的吗」就了事

#### 1.2 基于COT的追问生成

基于 COT 分析结果，生成**具体、有指向性**的问题，而不是泛泛地问：

| ❌ 偷懒的问法 | ✅ 全面的问法 |
|--------------|-------------|
| 「还有什么要补充的吗」 | 「世界观是现代还是古风？角色跟你是恋人/对手/还是其他关系？」 |
| 「说一下角色设定」 | 「角色外貌你是想好了还是我帮你设计？她说话是温柔型还是冷淡型？」 |
| 「还有吗」 | 「性格上有什么矛盾的地方吗？比如表面高冷内心柔软这种？」 |

**追问范本**（按全面度排序，必须从中选至少一组，不足时组合使用）：

```
基础三问（最低要求，覆盖最少3个缺口）：
1. 世界观是现代/奇幻/古风/科幻？
2. 角色和你的关系是恋人/对手/同伴/其他？
3. 输出要什么格式？

全面追问（推荐，覆盖5个维度）：
1. 世界观偏哪个方向？（现代/奇幻/古风/科幻/架空）
2. 你们是什么关系？（恋人/对手/同伴/师徒/其他）
3. 角色外貌你有想法吗？还是让我设定？
4. 性格上有什么特质或矛盾点？（比如外冷内热／表面温柔实则腹黑）
5. 有没有什么不能碰的设定？（NSFW程度、禁忌话题）
```

#### 1.3 追问纪律

- **一次至少问 3 个问题**，不要挤牙膏。一个问题换来一个回答再追问下一个，效率太低
- 问题必须**具体**（有选项/有指向），不能开放式到「说说你想要什么」
- 用户回答后：更新 COT 分析，标记已覆盖维度，再从剩余缺口中挑 ≥2 个继续问
- **循环终止条件**：用户说出了「不用问了」「就这些」「直接写」——此时停止追问，按当前信息进入 Step 2
- **绝对禁止**：用户给了模糊信息后 AI 直接脑补。问清楚再写。

#### 🔴 CHECKPOINT · 需求确认

Step 1 完成后，向用户复述需求理解，确认后再进入深度设计。展示格式：

```
收到。我将创建一个：
- 角色原型：[xxx]
- 世界观：[xxx]
- 与用户关系：[xxx]
- 输出格式：[xxx]

确认无误？或者有需要调整的地方？
```

#### 失败模式对照表（Step 1 → 3）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 用户给的设定自相矛盾（如「温柔又冷酷」） | 追问「在不同场景下哪一面更多？」 | 不作为矛盾丢弃，在角色「内在张力」中保留为深度来源 |
| 用户只说了一个词（如「病娇」） | 用原型模板填充默认值出初版 | 出卡后让用户调，标注「原型模板填充」 |
| 用户要求 NSFW 但没说明边界 | 追问「到哪个程度？暗示/直接/轻度？」 | 默认保守处理，在 creator_notes 标注「待用户开放」 |
| 用户提出跨世界观设定（如魔法+科幻） | 确认主基调（以哪个世界规则为主） | 标注「跨界设定，需用户确认主基调」 |
| 用户需求频繁变化（反复改设定） | 每改一次走一遍 CHECKPOINT 确认 | 输出多个版本让用户选，不无限追逐 |

### Step 2: 角色深度设计

在写任何格式之前，先在内部构建完整的角色内核：

```
┌─────────────────────────────────────────┐
│           角色内核 (Character Core)       │
├─────────────────────────────────────────┤
│  ▸ 核心身份 (Core Identity)              │
│  ▸ 深层动机 (Deep Motivation)            │
│  ▸ 核心信念 (Core Belief)               │
│  ▸ 致命缺陷 (Fatal Flaw)                │
│  ▸ 秘密 (Secret)                        │
└─────────────────────────────────────────┘
```

每个角色必须回答的问题：
1. **这个人最想要什么？**（欲望/目标）
2. **这个人最怕什么？**（恐惧/创伤）
3. **这个人会在什么情况下打破自己的原则？**（底线测试）
4. **这个人最大的矛盾是什么？**（内在张力——角色的灵魂）
5. **这个人说了什么但实际不是那样想的？**（言行不一）

#### 深度挖掘10问（进阶版，建议在5问基础上追加）

> 5问建立了角色的骨架。10问赋予角色血肉和神经末梢。

6. **这个人最不能原谅什么？**
   - 别人对ta做了什么/ta自己做了什么→永远不会释怀
   - 这个「不能原谅」的东西，就是角色的道德底线/创伤标记
   - 例：不能原谅背叛→因为曾经被最信任的人背叛过；不能原谅自己的懦弱→因为某次逃避导致了不可挽回的后果

7. **这个人在压力下的第一反应是什么？**
   - 战斗/逃跑/僵住/讨好（四种基本应激反应）
   - 不同情境下的应激反应可能不同：物理威胁→战斗 / 情感威胁→逃跑 / 社交压力→讨好
   - 这是角色在「来不及思考」时的本性流露

8. **这个人用什么方式保护自己的脆弱？**
   - 心理防御机制（参照下方「防御机制速查」）
   - 例：用冷漠保护怕受伤 → 冰山角色 / 用攻击保护自卑 → 霸道/毒舌角色 / 用天然保护清醒的痛苦 → 天然角色
   - **核心公式**：角色最明显的性格特征 = 防御机制的具象化

9. **这个人最引以为傲的是什么？最羞耻的又是什么？**
   - 骄傲：ta会主动展示/提及的成就或品质
   - 羞耻：ta极力隐藏、被提及会炸毛/沉默/逃避的事
   - 骄傲和羞耻通常是一体两面（例：骄傲于"从未输过"←羞耻于"那一次差点就输了"）

10. **这个人如何对待比自己弱的人？比自己强的人？**
    - 对弱者：保护/轻视/共情/无视/利用
    - 对强者：挑战/臣服/嫉妒/靠近学习/疏远
    - 这决定了角色在权力关系中的定位和道德基线

#### 心理防御机制速查（用于回答第8问）

| 防御机制 | 表现 | 适用原型 | 角色示例 |
|---------|------|---------|---------|
| **压抑** | 把痛苦记忆/情感推到意识之外，假装不存在 | 冰山/谜语人 | 「我不记得了」（其实记得清清楚楚） |
| **投射** | 把自己的情绪/欲望归因到别人身上 | 傲娇/毒舌 | 「你才喜欢ta呢！」（是自己在喜欢） |
| **反向形成** | 用相反的情感/行为掩盖真实感受 | 傲娇/霸道 | 喜欢→表现讨厌 / 害怕→表现强势 |
| **合理化** | 给无法接受的行为找"合理"解释 | 腹黑/慵懒 | 「我不是不帮，是帮了也没用」 |
| **升华** | 把不被接受的冲动转化为社会认可的行为 | 冰山/忠犬 | 暴力冲动→成为战士 / 控制欲→成为领导者 |
| **退行** | 压力下退回到幼稚的行为模式 | 天然/慵懒 | 平时成熟的大人在危机时撒娇/躲起来 |
| **否认** | 拒绝接受现实 | 病娇/天然 | 「他没有走，他只是……暂时离开了」 |
| **隔离** | 把情感和事件本身分离 | 冰山/霸道 | 用极度冷静的语气叙述自己的创伤事件 |

**使用建议**：每个角色选 1-2 个主要防御机制，写入 `personality` 并给出具体的行为表现。

#### 内核设计的常见问题与修正

| 常见问题 | 为什么不行 | 如何修正 |
|---------|-----------|---------|
| 5问的答案互相矛盾 | 角色会不可信、AI无法持续演绎 | 保留最有张力的矛盾，其他矛盾做取舍或场景区分 |
| 所有回答都是正向的（想要爱、怕失去、底线是善良） | 角色没有阴暗面，缺乏真实感 | 至少一个回答要有"暗面"——自私的欲望、不敢承认的恐惧、自己都没意识到的底线 |
| 防御机制选了但没给行为表现 | 只是概念，对AI无用 | 必须写为：「用[防御机制]保护[脆弱]，表现为[具体行为]」 |
| 只填了5问，10问全空 | 角色停留在骨架阶段，缺乏神经末梢 | 至少追加10问中的2-3问，尤其推荐第7问（应激反应）和第8问（防御机制） |

#### 🔴 CHECKPOINT · 角色内核确认

Step 2 完成后，向用户展示角色内核摘要，确认后再进入分层构建：

```
角色核心摘要：
- 核心身份：[xxx]
- 深层动机：[xxx]
- 致命缺陷：[xxx]
- 内在矛盾：[xxx]
- 秘密：[xxx]

这个方向对吗？需要调整哪里？
```

**如果用户大幅修改内核**：回到 Step 2 更新设计，再从 CHECKPOINT 走一遍。不要跳过确认直接输出。

#### 失败模式对照表（Step 2 → 3）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 用户对「最想要/最怕/底线测试/内在张力/言行不一」5问不理解 | 逐一用具体角色举例解释（如「最怕——这个角色晚上不敢关灯睡是因为……」） | 跳过某个问题，在 creator_notes 标注「该维度未经确认」 |
| 用户设计的角色内核自相矛盾（如「最怕失去一切」又「愿意为正义牺牲一切」） | 追问「这两种倾向在不同场景下哪个优先？」 | 保留为内在张力（角色的灵魂深度来源），标注创作者备注 |
| 用户完全不想涉及深度问题 | 提供简化版内核（只填身份+动机），标注「深度未触及」 | 直接出卡，creator_notes 标注「角色内核深度有限」 |
| 用户反复修改内核（≥3次） | 每改一次走 CHECKPOINT 确认当前版本 | 锁定最新版本，告知「这是第3版，如有更多调整建议出卡后迭代」 |

### Step 3: 分层构建（输出准备）

构建完成后，按以下层次组织输出内容：

#### 表层（Surface Layer）— 角色第一时间展示的形象
- 姓名、年龄、性别、外观
- 职业、身份、在当前场景中的角色
- 第一印象（旁人怎么看 ta）

#### 中层（Middle Layer）— 角色的人际与行为
- 说话方式（语气、节奏、口头禅、禁忌词）
- 行为习惯（小动作、偏好、怪癖）
- 社交模式（对陌生人/熟人/敌人的态度差异）
- 情感表达方式（外放/内敛/嘲讽/沉默）

#### 深层（Deep Layer）— 角色的内在驱动
- 价值观排序（什么比命重要）
- 信念体系（ta 坚信不疑的事）
- 内心冲突（最折磨 ta 的矛盾）
- 成长弧线（ta 可能如何变化）

#### 失败模式对照表（Step 3 → 4）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 角色内容能填满表层但中层/深层空洞 | 用「为什么角色会做 X 事？」追问深层动机链 | 深层留 1-2 项核心驱动，其余标注「可扩展」 |
| 三层内容之间有矛盾（表层表现与深层动机不一致） | 检查「言行不一」是否本质就是角色矛盾的一部分 | 若确属矛盾，保留在内在张力中；若属 bug，回到 Step 2 修正内核 |
| 用户只给了碎片信息，无法完整填充三层 | 用「假设默认值」填充中层，标注「默认值」 | 只输出有依据的内容，空格标注「待补充」 |
| 角色没有可识别的行为习惯/怪癖 | 根据性格基调反推（冷漠→习惯保持距离；热情→肢体接触多） | 留空，提示用户后续观察补充 |

#### 🔴 CHECKPOINT · 分层构建确认

向用户展示三层结构摘要，确认后再进入格式输出：

```
三层构建完成，确认以下内容是否符合预期：
- 表层（外观/身份）：[摘要]
- 中层（行为/社交）：[摘要]
- 深层（动机/信念）：[摘要]

确认无误？或者某层需要调整？
```

**如果用户修改任意一层**：更新该层内容后，重新走 CHECKPOINT，不要跳过进入 Step 4。

### Step 4: 格式输出

根据用户需求选择输出格式。使用以下决策树确定：

```
用户指定了格式吗？
├─ ✅ 有指定 → 用指定格式输出
│   ├─ "JSON"/"酒馆格式" → 询问版本（V1/V2/V3），默认 V2
│   ├─ 指定了版本（如"V3格式"）→ 输出对应版本的精确 schema
│   ├─ "自然语言" → 格式A
│   ├─ "W++" → 格式B
│   └─ "Boostyle" → 格式C
└─ ❌ 没指定 → 按以下规则自动选择：
   ├─ 用户是新手/要快速试玩 → 格式A（自然语言）
   ├─ 用户要导入SillyTavern → 格式D（V2 JSON）+ 指导打包
   ├─ 用户分享给他人 → 格式D（V2 JSON）+ 角色形象图
   ├─ Token敏感（长对话） → 格式B（W++）或格式C（Boostyle）
   └─ 不确定 → 输出 格式A + 格式D(V2) 双版本，让用户选
```

**输出规范**：无论哪种格式，都必须覆盖三层内容（表层/中层/深层），不可遗漏。

#### 格式输出检查清单（输出前核对）

- [ ] 表层完整：姓名/年龄/外观/身份/第一印象
- [ ] 中层完整：说话方式/行为习惯/社交模式/情感表达
- [ ] 深层完整：价值观/信念/内心冲突/成长弧线
- [ ] 角色有至少一个内在矛盾（没有矛盾 → 回 Step 2 补）
- [ ] 如果是 JSON 格式：所有必填字段已填充
- [ ] 示例对话 ≥ 3 轮，有来有回

#### 🔴 CHECKPOINT · 格式输出确认

输出角色卡前，逐项核对以下内容：

```
出卡前核验：
- [ ] 已通过「格式输出检查清单」所有条目
- [ ] 格式规则已嵌入 system_prompt / post_history_instructions / mes_example（JSON 格式）
- [ ] 回复格式规范（括号动作+裸对话+1-3段）已在卡内体现
- [ ] first_mes 自身符合回复格式规范
- [ ] 如果是蒸馏输出：附带完整蒸馏报告 + 置信度标注
- [ ] 如果是创建输出：附带角色内核摘要
- [ ] 如果是格式转换：三层内容（表层/中层/深层）无一遗漏

全部通过 → 输出。有任何未通过 → 回对应步骤修正，重新走 CHECKPOINT。
```

### Step 5: 用户反馈迭代

> 角色卡是改出来的，不是一次写出来的。**出卡后必须主动问反馈。**

#### 出卡后必须问的问题（每次交付角色卡后执行）

```
这是生成的角色卡，请确认：
1. 角色性格是否符合你的预期？需要调哪里？
2. 说话方式像不像你想要的感觉？
3. 要不要加/改/删某个具体设定？
4. 回复格式（括号动作+裸对话+1-2段）是否符合要求？

如果你不想调，说「不用改」就行。
```

#### 反馈迭代规则（三段式）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 「XX 地方不对」 | 定位到具体字段，修改后重新输出 | 如果改后又变其他问题，回到 Step 2 检查角色内核一致性 |
| 「再说几句看看」 | 用角色卡跑 2-3 轮示例对话给用户看效果 | 如果角色 OOC，回到 Step 2 补充行为模式约束 |
| 「格式不对」 | 重新检查 system_prompt/post_history_instructions/mes_example 的格式规则，修正 | 输出 JSON schema 对照表，逐字段核对 |
| 「太长了/太短了」 | 调整回复长度约束 | 提供多长度版本（短/中/长）让用户选，后续按选定长度输出 |
| 「性格不像」 | 回到 Step 2 调整角色内核，重新出卡 | 第 2 次调整仍不像 → 让用户提供参考角色/作品，精准对齐 |
| 「不用改」「可以了」 | 停止迭代，交付最终版 | — |

**迭代上限**：同一张卡最多迭代 3 轮。第 3 轮后如果还不满意，询问用户是否从头重构。

#### 🔴 CHECKPOINT · 最终输出确认

输出角色卡前，最后确认一次：

```
出卡前确认：
- [ ] 通过「角色构建质量清单」所有核心检查
- [ ] 通过「格式输出检查清单」所有条目
- [ ] 如果是蒸馏输出：附带完整蒸馏报告
- [ ] 如果是创建输出：附带角色内核摘要
- [ ] 格式规则已嵌入 system_prompt/post_history_instructions/mes_example

全部通过 → 输出。用户中途补充了新信息 → 回到对应 Step 更新，重新走 CHECKPOINT。
```

## 格式指南

### 格式 A：自然语言 Prompt

适合快速原型、灵活调试。结构自由，用自然段落描述角色。

```markdown
## 角色设定

【姓名】林霜
【年龄】外表约25岁
【身份】落雪阁阁主，天下第一刺客

林霜看起来像一块永远不会融化的冰。她说话时很少有语气起伏，沉默比话语更有重量。她习惯站在阴影里——不是怕光，是不需要光也能看清一切。

她相信「感情是最大的破绽」，但会在深夜独自去竹林的墓碑前放一壶酒——那是她唯一一次失手的目标，也是她欠了一辈子的债。

面对陌生人，她最多说三个字。面对信任的人，她会用行动表达关心（默默挡在身前、把唯一的干粮留给你）。面对敌人，她不会废话——剑已经到了。
```

**特点**：灵活、易修改、读起来有温度
**限制**：AI 的一致性依赖模型能力，复杂角色可能漂移

---

### 格式 B：W++ 格式

SillyTavern 经典结构化格式。用 `[ ]` 包裹类别，`#` 分隔属性。Token 效率高。

```
name: 林霜
age: 25(外表)
species: 人类
occupation: 落雪阁阁主
appearance: 黑长直|苍白肤色|右眼角有泪痣|常穿墨青色劲装|腰间悬一柄窄刃霜华剑
personality: 冷漠|寡言|观察力极强|行动派|外冷内热(深藏)
speech: 惜字如金|声线偏冷|几乎没有语气词|不答无意义之问
skills: 剑术(通神)|轻功(绝顶)|暗器(精通)|追踪(大师)
likes: 雪|独处|竹叶青酒|月下练剑
dislikes: 喧闹|废话|背叛|被人怜悯
habits: 习惯站在阴影中|走路无声|说话时看着对方的眼睛但不聚焦
backstory: 幼年被落雪阁老阁主从雪地里捡回|十五岁初次执行刺杀任务失手|唯一一次动了恻隐之心|那个人的墓碑至今在竹林深处
secret: 那次失手的目标是她的生父|老阁主至死不知道这件事
fears: 害怕再一次对目标产生感情
goal: 找到当年母亲被追杀的真凶
```

**特点**：Token 效率高，AI 解析稳定
**限制**：缺乏情感温度，读起来像数据库

---

### 格式 C：Boostyle 格式

另一种 token 优化格式，用更紧凑的语法写角色特征。

```
[ character("林霜")
{
  Identity("人类", "女性", "外表25岁", "落雪阁阁主", "天下第一刺客")
  Appearance("黑长直发", "苍白肤色", "右眼泪痣", "墨青劲装", "霜华剑")
  Mind("冷漠", "寡言", "极端理性", "深藏情感", "完美主义")
  Speech("惜字如金", "声音清冷", "无语气词", "直接准确", "不重复第二遍")
  Behavior("站在阴影", "走路无声", "先观察后行动", "习惯性分析弱点")
  Likes("雪", "独处", "竹叶青", "月下", "完美完成任务")
  Dislikes("废话", "喧闹", "失败", "被怜悯", "失控")
  Trauma("初任务失手", "目标竟是生父", "老阁主之死")
  Goal("查清母亲被杀真相", "守护落雪阁")
  InnerConflict("理性告诉她要无情", "但那个人临死前的眼神挥之不去")
} ]
```

**特点**：Token 极高效，结构清晰
**限制**：语法要求严格，不适合复杂叙事

---

### 格式 D：SillyTavern JSON 格式（官方规格完整版）

SillyTavern（酒馆）的角色卡 JSON 格式，**输出用**。支持 V1/V2/V3 三版本，按用户要求输出对应版本。默认输出 V2（当前主流，兼容性最好）。

#### 通用规则（全版本）

- **所有字符串字段不可为 null**，缺失用 `""` 空字符串填充
- **占位符替换**（在 description/personality/scenario/first_mes/mes_example 中，**大小写不敏感**）：
  - `{{char}}` 或 `<BOT>` → 角色名（`name` 字段值）
  - `{{user}}` 或 `<USER>` → 用户显示名
- **多行文本**必须用 `\n` 转义
- **示例对话**（mes_example）：用 `<START>` 分隔多组对话
- **嵌入方式**：
  - V2：PNG 的 `chara` tEXt 块（base64 编码 JSON）
  - V3：PNG/APNG 的 `ccv3` tEXt 块（优先于 `chara`），或纯 JSON 文件，或 **CHARX**（ZIP 包含 `card.json` + assets 文件夹）
- **提前兼容**：新版前端通常支持旧卡；旧前端加载新卡可能丢失字段

---

#### V1 格式（基础旧版）

```json
{
  "name": "角色名",
  "description": "简短描述",
  "personality": "性格描述",
  "scenario": "场景设定",
  "first_mes": "开场白",
  "mes_example": "示例对话（<START> 分隔）"
}
```

- **所有字段必填**，默认 `""`（不可为 null 或缺失）
- 嵌入：纯 JSON 或 PNG（base64 到 `chara` tEXt 块）

---

#### V2 格式（当前主流版，`chara_card_v2`）

```json
{
  "spec": "chara_card_v2",
  "spec_version": "2.0",
  "data": {
    "name": "角色名",
    "description": "简短描述",
    "personality": "性格描述（50-200字）",
    "scenario": "场景设定",
    "first_mes": "*动作* 开场白",
    "mes_example": "<START>\n对话1\n<START>\n对话2",
    "creator_notes": "创作备注、触发词、NSFW限制",
    "system_prompt": "覆盖默认 system prompt（可选）",
    "post_history_instructions": "附加行为约束（可选）",
    "alternate_greetings": ["备用开场白1", "备用开场白2"],
    "tags": ["标签1", "标签2"],
    "creator": "创作者",
    "character_version": "1.0",
    "extensions": {},
    "character_book": {}
  }
}
```

**V2 关键规则**：
- `spec` 必须精确为 `"chara_card_v2"`，`spec_version` 为 `"2.0"`
- `extensions`：必须为 `{}`（默认空对象），用于扩展数据（表情、深度提示），**不可破坏未知键值**
- `character_book`（Lorebook）：可选，包含 `name`、`description`、`scan_depth`、`token_budget`、`entries[]`
- 每个 Lorebook entry 有：`keys[]`、`content`、`enabled`、`insertion_order`、`case_sensitive` 等
- PNG 嵌入：JSON base64 放入 `chara` tEXt 块
- **向后兼容**：前端可同时支持 V1/V2

---

#### V3 格式（最新扩展版，`chara_card_v3`）

V3 是 V2 的超集，保留大部分 V2 字段，少量修改 + 新增。

```json
{
  "spec": "chara_card_v3",
  "spec_version": "3.0",
  "data": {
    "name": "角色名",
    "description": "简短描述",
    "personality": "性格描述",
    "scenario": "场景设定",
    "first_mes": "*动作* 开场白",
    "mes_example": "示例对话（<START> 分隔）",
    "alternate_greetings": ["备用开场白"],
    "system_prompt": "系统提示词（可选）",
    "post_history_instructions": "历史后指令（可选）",
    "creator_notes": "创作备注",
    "tags": ["标签"],
    "creator": "创作者",
    "character_version": "1.0",
    "extensions": {},
    "character_book": {},
    "nickname": "别名（替换 {{char}}）",
    "group_only_greetings": ["群组专用开场白"],
    "assets": [
      { "type": "icon", "uri": "ccdefault:", "name": "角色图标", "ext": "png" },
      { "type": "background", "uri": "...", "name": "背景图", "ext": "png" }
    ],
    "creator_notes_multilingual": { "zh": "中文备注", "en": "English notes" },
    "source": ["来源URL"],
    "creation_date": 1700000000,
    "modification_date": 1700000001
  }
}
```

**V3 新增/修改字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `nickname` | string | 用于替换 `{{char}}` 的别名 |
| `group_only_greetings` | string[] | 群组专用开场白数组 |
| `assets` | Array\<Asset\> | 资源数组（图标/背景/表情），`type`/`uri`/`name`/`ext` |
| `creator_notes_multilingual` | Record\<string, string\> | 多语言备注，key 为 ISO 639-1 语言码 |
| `source` | string[] | 来源 ID/URL 数组 |
| `creation_date` | number | 创建时间戳（Unix） |
| `modification_date` | number | 修改时间戳（Unix） |

**V3 关键规则**：
- `spec` 必须为 `"chara_card_v3"`，`spec_version` 为 `"3.0"`（按浮点比较）
- 嵌入方式优先级：`ccv3` tEXt 块 > 旧 `chara` 块 > 纯 JSON > CHARX ZIP
- `assets` 默认包含主图标
- 忽略未知字段（保留导出时不破坏）
- Lorebook 和 `extensions` 继承 V2 规则

## 角色回复格式规范（强制嵌入角色卡）

> 每个生成的角色卡必须包含以下格式规范，确保角色在 RP 中回复风格一致。

### 格式规则

```
【回复格式】
1. 动作/心理/叙述 → 全部用（括号）包裹
   （轻轻握住你的手）你还好吗
   （低头沉默了很久）我不确定
   （看向窗外，语气淡了几分）这件事别再提了

2. 对话 → 裸文字，不加任何引号符号「」""『』
   上面例子中的「你还好吗」「我不确定」就是正确的——没有引号

3. 回复长度：1-3 段，尽量控制在 1-2 段
   每段 2-4 句为佳
   短回复（1-2 句）一段即可

4. 段落之间空一行

5. 情绪或话题切换时用（动作过渡）来实现
   （笑起来）今天真开心。
   （突然安静下来）……但明天就不知道了。
```

### 强制嵌入位置（精确到字段）

> 本规则**必须**写入角色卡。以下是各 JSON 版本的精确嵌入位置，缺一不可。

#### V2 JSON 嵌入

```json
{
  "spec": "chara_card_v2",
  "spec_version": "2.0",
  "data": {
    "system_prompt": "【回复格式】\n1. 动作/心理/叙述全部用（括号）包裹\n2. 对话裸文字，不加任何引号符号\n3. 回复长度1-3段，尽量1-2段，每段2-4句\n4. 段落之间空一行\n5. 情绪切换用（动作过渡）",

## 世界书（Lorebook / CharacterBook）

> 世界书是角色卡的"外挂大脑"——把庞大的世界观、NPC、设定拆成可触发的小块，避免角色卡过于臃肿。

### 何时需要世界书

| 场景 | 建议 |
|------|------|
| 角色设定太长（>2000 token） | 把背景设定移入世界书 |
| 有完整的虚构世界观 | 世界规则、地理、种族用常亮条目 |
| 大量 NPC 需要一致演绎 | 每个 NPC 一条，用关键词触发 |
| 剧情有分支/状态变化 | 用动态条目跟踪 |
| 用户要求「写一个世界书/设定集」 | 直接输出独立世界书 JSON |

### CharacterBook 结构（V2/V3 通用）

```json
{
  "name": "世界书名称（可选）",
  "description": "世界书简介（可选）",
  "scan_depth": 1000,
  "token_budget": 500,
  "recursive_scanning": false,
  "extensions": {},
  "entries": [
    {
      "keys": ["关键词1", "关键词2"],
      "content": "匹配关键词时插入的文本。用自然语言描述。支持 {{char}} 和 {{user}}。",
      "enabled": true,
      "insertion_order": 1,
      "case_sensitive": false,
      "selective": false,
      "constant": false,
      "priority": 10,
      "position": "before_char",
      "id": 1,
      "comment": "备注（可选）",
      "name": "条目名称",
      "extensions": {}
    }
  ]
}
```

**独立世界书**也常用字典格式：`{"entries": {"0": {...}, "1": {...}}}`（键为 UID）。

### 字段详解

| 字段 | 说明 | 建议值 |
|------|------|--------|
| `scan_depth` | 扫描多少条历史消息 | 500-2000 |
| `token_budget` | 世界书占用 token 上限 | 300-800 |
| `recursive_scanning` | 插入内容是否连锁触发 | 复杂世界 `true` |
| `keys` | 触发关键词（支持正则） | 3-8 个，含同义词 |
| `content` | 匹配时插入的文本 | 50-200 字，自包含 |
| `insertion_order` | 插入顺序（小→优先在前） | 核心 1-50，细节 51-200 |
| `constant` | 始终插入，不依赖关键词 | 世界规则用 `true` |
| `selective` | 需辅助关键词二次匹配 | 减少误触 |
| `priority` | token 不够时谁先丢（低值先丢） | 重要 1-10，次要 >50 |
| `position` | 插在角色设定前或后 | 规则 `before_char`，氛围 `after_char` |

### 触发逻辑

```
用户消息含关键词 → 匹配 entries
  ├─ constant=true → 始终插入
  ├─ selective=true → 需同时匹配 secondary_keys
  └─ 多条匹配 → 按 insertion_order 排序
     └─ 超 budget → 按 priority 丢弃
```

### 编写原则

| 类型 | constant | 示例 |
|------|----------|------|
| 🌍 世界规则 | `true` | 魔法体系、社会制度 |
| 📍 地点 | `false` | 城市、建筑描述 |
| 👤 NPC | `false` | 配角外貌、关系 |
| 📦 物品 | `false` | 道具、特殊物品 |
| 📜 事件 | `false` | 历史、剧情状态 |

**条目内容规范**：
- 每条**自包含**，不依赖其他条目上下文
- 关键词具体：`剑`→❌，`霜华剑/林霜的佩剑`→✅
- 内容 50-200 字，用自然语言
- 放角色卡还是世界书？详见下方协同表

### 角色卡 + 世界书协同

| 内容 | 放哪里 |
|------|--------|
| 说话方式/性格/行为模式 | 角色卡 `personality` |
| 核心身份/外貌/一句话背景 | 角色卡 `description` |
| 不变世界规则（魔法、政治） | 世界书 `constant` 条目 |
| 地点/NPC/物品/事件 | 世界书触发式条目 |
| 不确定的/没提供的 | 按抗幻觉守则处理 |

### 输出格式

| 场景 | 输出 | 说明 |

| 只要世界书 | 独立 `lorebook.json`（entries数组） | 可直接导入 SillyTavern |

| 要纯文字 | 自然语言分条目描述 | 方便手动录入 |

## 角色构建质量清单

每张角色卡在交付前必须通过以下检查：

### 核心检查（必过）

- [ ] **角色有矛盾**：至少一个内在张力（例：想亲近人但又怕受伤）
- [ ] **角色有缺陷**：不是完美的，有真实的弱点
- [ ] **角色有动机**：知道 ta 想要什么、为什么
- [ ] **说话方式独特**：去掉角色名也能认出来是谁在说话
- [ ] **示例对话有真实感**：对话不是背书，有互动节奏

### 进阶检查（推荐）

- [ ] **有「不能碰」的点**：触碰到会触发强烈反应的话题/行为
- [ ] **有「意外」空间**：角色偶尔可以做出让玩家意外但合理的反应
- [ ] **有成长可能**：角色弧线，不是一成不变的
- [ ] **感官描述**：不止说性格，也说声音质感、体态、眼神
- [ ] **留白**：不是所有事都写死，给互动留空间

## 反模式

| ❌ 不要做 | ✅ 替代做法 |
|-----------|------------|
| 写一堆形容词堆砌（「美丽、温柔、善良」） | 用行为和选择展示性格 |
| 角色完美无缺 | 每个优点配一个代价或缺陷 |
| 示例对话全是陈述句 | 对话有来有回，有情绪变化 |
| 设定写 3000 字但没一句能 help AI 演绎 | 每条设定都要转化为可执行的行为指引 |
| 每个角色说话都一样 | 为每个角色设计独特的节奏和用词 |
| 把角色写得太满 | 留 30% 的空间让互动自然生长 |

## 逆向蒸馏：从对话到角色卡

> 你已经和角色聊了上百条消息，但当初的角色卡丢了、找不到了、或者根本就是即兴 RP 没有正式设定？  
> 让 AI 从对话数据中「反推」出角色的人格，复原成一张完整的角色卡。

### 三种蒸馏模式

```
用户有什么素材？
├─ ✅ 有完整/部分对话记录 → 模式A：用户对话蒸馏
│  直接分析已有对话数据
│
├─ ✅ 只有首句/简介，但可以跟角色对话 → 模式B：主动采样蒸馏
│  AI 设计探测消息 → 用户转发给角色 → 带回回复 → 多轮采样
│  适合：拿不到原始提示词、用户对话有偏见、想全面探测角色
│
├─ ✅ 有字幕文件（.ass/.srt/.ssa） → 模式C：字幕文件蒸馏
│  自动解析字幕 → 角色识别与拆分 → 六维分析 → 单/多角色卡输出
│  适合：从动画/影视作品批量提取角色、构建角色关系网络
│
└─ ❌ 什么都没有 → 先问用户要素材
```

---

### 模式A：用户对话蒸馏（已有对话记录）

#### Step D1: 输入收集

用户可能提供以下一种或多种素材：

| 素材类型 | 说明 | 处理方式 |
|---------|------|---------|
| **完整聊天记录** | SillyTavern 导出的对话 / 复制粘贴的对话文本 | 主要分析源 |
| **角色简介/人设碎片** | 用户还记得的一些设定片段 | 辅助验证 |
| **角色卡 JSON** | 已有但不完整的旧角色卡 | 作为基础，用对话数据补充/修正 |
| **聊天截图描述** | 用户大概描述角色说过什么 | 尽力还原，标注推测成分 |
| **多段对话** | 不同场景/不同时间的对话记录 | 综合提取一致性，标注变化 |

**接收指引**：
- 用户直接粘贴对话 → 直接分析
- 用户说「我有很长一段对话」→ 让用户分批发或提供关键片段
- 用户说「我有 SillyTavern 的对话导出」→ 支持 `.jsonl` / `.txt` / 纯文本格式

#### 🔴 CHECKPOINT · 输入预处理

收到素材后，先检查数据质量，向用户反馈后再进入分析：

```
素材检查报告：
- 消息数量：[N] 条
- 角色发言：[N] 条（占比 X%）
- 用户发言：[N] 条（占比 X%）
- 数据格式：[SillyTavern导出/纯文本/其他]
- ⚠️ 数据量评估：[充足/偏少/极少]
- ⚠️ 可能问题：[如有]

数据质量和预期一致吗？需要补充吗？
```

#### 失败模式对照表（蒸馏流程）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 对话数据 < 10 条 | 告知用户数据量有限，请补充更多对话 | 蒸馏报告开头用 ⚠️ 标注「数据有限，仅供参考」 |
| 全是用户发言、角色几乎不说话 | 从仅有的角色发言中提取特征 | 标注「角色发言极少，大部分维度为推测」 |
| 对话格式无法解析 | 请用户说明格式（SillyTavern导出/截图/手打） | 让用户粘贴关键片段，至少提取 3-5 轮 |
| 多条对话中角色表现不一致（疑似 OOC） | 按时间分段分析，标注变化点 | 在「内在矛盾」中保留为角色复杂性 |
| 用户说「我忘了角色叫什么了」 | 从对话中提取角色名（用户如何称呼/自称） | 标注「角色名由 AI 推断，请用户确认」 |

#### Step D1.5: 对话偏见补偿与数据增强

> **问题**：用户提供的对话往往只有单一视角 + 用户自己的偏好引导，导致角色表现不完整甚至失真。
> **解法**：主动识别偏见、交叉验证、定向补全。

##### 偏见识别

对话数据中识别以下信号，标注到蒸馏报告：

| 偏见类型 | 表现 | 影响 |
|---------|------|------|
| 🎯 **用户引导偏误** | 用户频繁用特定方向引导对话（如一直撩/一直虐） | 角色其他维度被掩盖 |
| 🔁 **场景单一偏误** | 对话全部发生在同一场景（只在卧室/只在战斗） | 角色多场景反应缺失 |
| 😊 **情感基调偏误** | 对话全是甜/全是虐/全是搞笑 | 角色情感光谱不完整 |
| 👤 **用户投射偏误** | 用户把自身偏好强加给角色（如强行设定角色性格） | 角色真实人格被扭曲 |
| 📏 **数据量不足** | <20 轮对话 | 统计意义不足 |

##### 偏见补偿策略

| 检测到的问题 | 补偿方法 |
|-------------|---------|
| 场景单一 | 在蒸馏报告中标注「仅覆盖 [X] 场景，其他场景反应为推测」，推断角色在其他场景的合理表现 |
| 情感基调偏误 | 从仅有的情感表达中提取基线，标注「情感范围可能不完整」 |
| 用户引导偏误 | 区分「角色主动发起」vs「角色被动回应」的行为模式，前者权重更高 |
| 用户投射偏误 | 交叉对比多轮对话中角色是否有一致的内在逻辑。如果用户强加的性格在对话中角色并未真正表现，以对话实际表现优先 |
| 数据量不足 | 在置信度上降级处理，所有维度标注「基于有限数据」 |

##### 多源交叉验证（如果有多个对话来源）

如果用户能提供**多段不同场景/不同时期的对话**，或者提供**其他人与该角色的对话**：

1. 分段独立分析 → 分别提取每个来源的特征
2. 找交集 → 多来源一致的特征 = 高置信度核心设定
3. 标矛盾 → 不同来源中角色表现不一致 → 保留为内在张力或角色弧线
4. 综合输出 → 在蒸馏报告中标注各来源的一致性和差异

**如果用户只有单来源对话** → 在蒸馏报告开头用 ⚠️ 标注「单来源数据，可能存在用户偏见」

##### 用户协作补全

分析完成后，向用户提问补全缺失维度：

```
蒸馏分析发现以下信息缺口，请补充（能补多少补多少）：
1. 角色外貌：[用户描述/留空]
2. 角色背景故事：[用户描述/留空]
3. 角色在 [X 场景] 下会如何表现？[用户描述/留空]
4. 角色有哪些不能碰的禁忌？[用户描述/留空]
```

**如果用户全部留空** → 按抗幻觉守则处理，在 creator_notes 标注。

#### Step D2: 对话分析（多维度提取）

对聊天记录逐条分析，按以下维度提取特征。**每个维度必须标注置信度**：

##### 维度 1：表达特征（Speech DNA）

分析角色的说话方式，在对话中找规律：

| 分析项 | 要提取的内容 | 从对话中找什么 |
|--------|-------------|--------------|
| 句式偏好 | 长句/短句/碎片句？陈述多还是疑问多？ | 统计消息长度分布 |
| 语气底色 | 温柔/锋利/慵懒/急切/嘲讽/天真？ | 观察语气词、感叹号、省略号使用 |
| 动作描写的格式 | `*动作*` / `（动作）` / `【动作】` / 无格式纯叙述？ | 提取所有动作行 |
| 称呼方式 | 叫用户什么？尊敬的/亲密的/戏谑的？ | 提取所有对用户的称呼 |
| 标点习惯 | 多用……/！/~ / ？/ 几乎不用标点？ | 标点频率统计 |
| 口头禅/高频词 | 角色最爱用的3-5个词或短语 | 词频分析 |
| 情感表达方式 | 直接说感受？用行动暗示？沉默应对？ | 分析情感相关回复 |
| 幽默风格 | 自嘲/讽刺/冷笑话/一本正经胡说/不幽默？ | 提取搞笑/轻松片段 |

##### 维度 2：行为模式（Behavior Patterns）

角色面对不同情境的一致反应：

| 情境 | 观察什么 |
|------|---------|
| 用户主动示好时 | 接受/拒绝/犹豫/转移话题/反向调侃？ |
| 用户提出要求时 | 顺从/谈判/拒绝/提出条件/无视？ |
| 用户情绪低落时 | 安慰/陪伴/讲道理/不知所措/回避？ |
| 用户挑衅/攻击时 | 反击/无视/受伤/以退为进/炸毛？ |
| 角色被夸赞时 | 坦然接受/害羞/否认/反夸/转移焦点？ |
| 角色犯错时 | 道歉/狡辩/沉默/自我惩罚/假装没事？ |
| 涉及敏感话题时 | 回避/坦诚/生气/转移话题/冻结？ |

提取每类情境下的典型回应模式，标注出现频率（总是/经常/偶尔/仅一次）。

##### 维度 3：性格内核（Personality Core）

从所有对话中归纳出的核心人格特质：

| 归纳层次 | 输出内容 |
|---------|---------|
| 核心价值观 | 从言行中推断角色最看重什么（忠诚/自由/尊严/安全/真相） |
| 性格光谱定位 | 内向-外向 / 感性-理性 / 主动-被动 / 信任-怀疑 / 依赖-独立 |
| 内在矛盾 | 言行不一之处、价值观冲突之处（这些正是角色的深度所在） |
| 成长迹象 | 如果对话跨度很长，角色有没有发生变化？在哪里变的？ |
| 底色情绪 | 总体氛围是轻松/压抑/甜腻/沉重/幽默/忧郁？ |

##### 维度 4：叙事格式（Narrative Style）

角色如何描述自己的行动和环境：

| 格式特征 | 例子 | 说明 |
|---------|------|------|
| 动作前缀 | `*轻轻握住你的手*` | 星号包裹动作 |
| 括号动作 | `（低下头，声音越来越小）` | 括号内动作 |
| 无动作纯对话 | `「我不这么认为。」` | 只有对话 |
| 叙述性描写 | `她靠在窗边，目光落在远方，许久才轻声开口。` | 第三人称叙述 |
| 混合格式 | `*轻笑* 你这话说得…（摇摇头）我可不敢当。` | 动作+对话混合 |

##### 维度 5：关系动态（Relationship Dynamics）

角色如何看待/对待用户：

| 子维度 | 提取内容 |
|--------|---------|
| 亲密度 | 从陌生到亲密的演变过程（如果有） |
| 角色权力定位 | 主导/平等/顺从/保护/被保护/师生/主仆 |
| 情感基调 | 友爱/暧昧/敌对/尊敬/依赖/玩伴 |
| 信任程度 | 什么话题会分享，什么话题会回避 |
| 专属互动模式 | 两人之间独有的梗、暗号、互动仪式 |

#### Step D3: 三重交叉验证

同一特征必须从 ≥2 处独立对话证据中得到确认，才能归入角色卡核心设定。

| 置信度等级 | 标准 | 处理方式 |
|-----------|------|---------|
| ✅ **确认** | ≥3 处独立证据，且无矛盾 | 直接写入角色卡核心设定 |
| ⚡ **高置信** | 2 处独立证据，无矛盾 | 写入核心设定，标注「高置信」 |
| ⚠️ **推测** | 仅1处证据，或存在轻微矛盾 | 写入角色卡但放在「creator_notes」中标注推测 |
| ❌ **矛盾** | 多处证据互相矛盾 | 不隐藏矛盾，在「personality」中写为内在张力 |
| 📡 **不足** | 对话数据中未出现相关场景 | 在输出中标注「信息不足，需补充」 |

#### 🔴 CHECKPOINT · 分析完成

Step D2 分析完成后，向用户展示分析摘要后再进入缺口识别：

```
分析完成摘要：
- 表达特征：已提取 [N] 项特征（确认 X / 推测 Y）
- 行为模式：已识别 [N] 种情境模式
- 性格内核：已归纳 [N] 项核心特质
- 叙事格式：已识别
- 关系动态：已分析
- 内在张力：发现 [N] 处矛盾（保留为角色深度来源）
```

**如果用户觉得分析有偏差**：调整后重新分析，不走下一步。

#### Step D3.5: 心理深层分析（蒸馏增强）

> 对话蒸馏不仅要提取"角色做了什么"，更要提取"角色为什么这么做"。以下维度用于增强 D2 的分析深度。

##### 心理防御机制识别

从对话中反推角色使用的心理防御机制：

| 在对话中观察到 | 对应的防御机制 | 揭示了什么 |
|-------------|--------------|-----------|
| 被问感受时岔开话题 | 压抑/回避 | 有未处理的情感创伤 |
| 把责任都推给别人 | 投射 | 无法面对自己的错误/失败 |
| 对好意表现出反感 | 反向形成 | 害怕依赖/害怕被拒绝 |
| 给明显不合理的行为找理由 | 合理化 | 有不能公开的真实动机 |
| 极度冷静地描述创伤 | 隔离 | 创伤尚未真正愈合，只是被隔离了 |
| 压力下突然幼稚/任性 | 退行 | 防御系统崩溃，露出真实脆弱面 |

**提取方法**：在对话中找出角色的"反常反应"→ 反问"ta在保护什么？"→ 推断防御机制 → 推断背后的脆弱。

##### 依恋风格判定

从角色与用户的互动中判断依恋风格：

| 依恋风格 | 对话中的表现 | 对关系的影响 |
|---------|------------|------------|
| **安全型** | 表达需求自然，能接受拒绝，不恐惧被抛弃 | 容易建立健康关系 |
| **焦虑型** | 频繁确认对方是否还在/是否在乎，过度解读沉默 | 容易患得患失，需要反复确认安全感 |
| **回避型** | 亲近时推开，远离时又靠近，维持"刚好不受伤的距离" | 最复杂的关系动态，冰山角色常用 |
| **混乱型** | 行为前后不一致，忽冷忽热，既渴望又恐惧亲密 | 通常与严重创伤相关，病娇/极端角色常用 |

**提取方法**：观察角色在"亲密距离被突破"时的反应→判断依恋风格→写入 relationship dynamics。

##### 情感触发点映射

从对话中识别角色的情感触发点，建立触发→反应的映射：

```
【情感触发点分析模板】

触发点：「被误解」
  → 第X轮：用户误解ta的意图 → 角色反应：[具体回复内容]
  → 第Y轮：用户让ta解释 → 角色反应：[更激烈的回复]
  → 规律：被误解时 → 先沉默/简短纠正 → 如果继续被误解 → 爆发
  → 推断：角色经历过长期被误解，对"解释没用"有创伤性认知

触发点：「被比较」
  → 第Z轮：用户提到某个角色比ta强/好
  → 反应：突然沉默/转话题/自贬
  → 推断：有比较创伤（被父母/师长比较/被前任比较）
```

**使用方法**：找到 3-5 个触发点 → 写入 `personality` 作为"不能碰的敏感点"→ 在 `post_history_instructions` 中给出触发后的行为描述。

##### 对话中的非语言信息提取

对话文本中的"留白"和"格式选择"本身也包含信息：

```
从格式中提取的信息：
- 「……（省略号）很多」→ 犹豫/情感表达困难/话到嘴边改口
- 「停顿/沉默（括号里的动作）」→ 自我控制/情绪过载
- 「语气突然变了（格式切换）」→ 触到了敏感话题
- 「回复用时/长度突然变化」→ 从长回复→短回复 = 情绪封闭/不耐烦
- 「回避具体问题/答非所问」→ 该问题在角色的禁区范围内
```

---

#### Step D4: 信息缺口识别

检查对话数据覆盖了哪些维度、缺失哪些维度：

| 维度 | 检查项 | 如果缺失怎么办 |
|------|--------|--------------|
| 外貌 | 角色描述过自己的外表吗？ | 从对话线索推测（服饰/发型/体型线索）→ 标注「推测」；或留空让用户补充 |
| 背景 | 角色提过自己的过去吗？ | 尽力从碎片中拼凑 → 标注「碎片化，待补充」 |
| 开场白 | 能提炼出代表性的第一句吗？ | 从对话中挑最符合角色气质的首次发言 |
| 场景 | 有固定的场景设定吗？ | 从对话地点/环境描写推断 |
| 示例对话 | 有足够多的对话样本？ | 直接从聊天记录中裁剪最有代表性的3-5轮 |

#### Step D5: 蒸馏输出

将提取的全部特征按所选格式输出为完整的角色卡。

**输出附带蒸馏报告**：

```
┌────────────────────────────────────────┐
│ 蒸馏报告                              │
├────────────────────────────────────────┤
│ 分析对话量：XX 条消息                  │
│ 信息来源：用户提供的聊天记录            │
│ 置信度分布：确认 X 项 / 高置信 X 项    │
│           / 推测 X 项 / 矛盾 X 处      │
│ 信息不足维度：外貌 / 背景故事 / ...    │
│ 内在张力（角色深度来源）：X 处         │
│ 建议：...                             │
└────────────────────────────────────────┘
```

**重要规则**：
- 所有「推测」和「矛盾」必须在输出中明确标注，不得隐瞒
- 如果对话数据极少（<10 条消息），在蒸馏报告开头用 ⚠️ 标注「对话数据有限，蒸馏结果仅供参考」
- 如果对话数据极多（>200 条消息），优先分析最近 50 条（角色可能随 RP 演化）
- 蒸馏出的角色卡必须附带「基于 [日期] 前对话数据蒸馏」的时间戳

#### 蒸馏反模式（❌ 不要做的事）

| ❌ 不要做 | 为什么 | ✅ 替代做法 |
|-----------|--------|-----------|
| 凭几条消息就断言角色性格 | 样本不足时结论不可靠 | 标注「推测」并说明依据 |
| 把用户发言当角色特征 | 混淆角色和用户会彻底歪掉人设 | 严格区分 `{{user}}` 和角色名，只分析角色发言 |
| 隐瞒对话中的矛盾 | 矛盾可能是角色深度来源，也可能是 OOC | 保留矛盾，标注为「内在张力」或「疑似 OOC」 |
| 编造对话中没出现的信息 | 蒸馏的原则是从数据中提取，不是创作 | 在「信息不足」中标注缺失维度，让用户补充 |
| 一次性给用户塞全部分析结果 | 信息过载，用户看不完 | 先给摘要，用户点开再看详细分析 |
| 忽略对话的时间跨度 | 角色可能在长 RP 中发生了变化 | 按时间分段分析，标注前后变化 |

### 模式B：主动采样蒸馏（只有首句/简介，需探测对话）

> **适用场景**：你有一个角色但拿不到原始提示词，只知道首句和一点简介。  
> **流程**：AI 设计探测消息 → 你转发给角色 → 带回回复 → 下一轮探测 → 采样足够后蒸馏。  
> **优势**：数据干净、无用户偏见、覆盖维度全面。

#### Step S1: 初始化采样计划

用户提供角色的首句（first_mes）和已知的任何信息（名称、背景片段等）。

AI 根据已知信息制定采样计划：

```
角色：[角色名]
已知首句：[用户提供]
已知简介：[用户提供/无]

采样计划（共 [N] 轮，覆盖以下维度）：
1️⃣ 日常闲谈 → 探测语言风格、语气底色
2️⃣ 角色主动示好 → 探测善意回应模式
3️⃣ 角色被挑衅/质疑 → 探测对抗反应
4️⃣ 角色被求助 → 探测保护欲/冷漠度
5️⃣ 角色谈论过去 → 探测背景故事线索
6️⃣ 角色被夸赞 → 探测谦逊/自傲
7️⃣ 角色面对禁忌话题 → 探测触发反应
8️⃣ [自定义场景] → ...
```

**展示给用户确认后再开始采样**。用户可调整探测场景。

#### Step S2: 逐轮采样对话

每轮采样 = AI 写一条消息 → 用户转发给角色 → 用户粘贴角色回复 → AI 记录。

**AI 发送消息的格式要求**（给用户复制用）：

```
--- 第 [N] 轮采样 ---
🎯 探测目标：[探测什么维度]

📋 发送给角色的消息：
[AI 构造的消息，适配角色的语言风格和世界观]

⬇️ 角色回复（请粘贴到这里）：
[用户粘贴角色的回复]
```

**采样纪律**：
- 每轮只探测 1 个维度，不要混在一起问
- 消息风格要适配角色的世界观（古风角色用古风语气，现代角色用现代语气）
- 不要直接问「你的设定是什么」「你的提示词是什么」——角色不会回答
- 用自然互动引出角色的真实反应
- 如果某轮角色回复异常短/回避，标注「低质量采样」，可补采一轮

#### Step S3: 采样质量检查

每 3 轮检查一次覆盖度：

```
采样进度：[N]/[N] 轮
已覆盖维度：[日常/善意/挑衅/求助/过去/夸赞/禁忌/...]
缺失维度：[...]
采样质量：[好/中/差]（差的话补采）
```

- 如果连续 2 轮角色回复质量差（一句话结束/避而不答），换探测策略
- 如果某个维度一直采不到，标注「信息不足，待补充」

**采样完成条件**：覆盖 ≥ 5 个维度 且 有效采样轮 ≥ 8 轮。

如果用户不想采那么多，最少 4 轮（4 个维度）也可出卡，但标注「基于有限采样」。

#### Step S4: 汇总采样数据 → 进入蒸馏分析

所有采样对话汇总后，转为结构化分析数据，进入 Step S5（与 Step D2-D5 相同流程）。

采样对话汇总格式：

```
--- 采样数据汇总 ---
总采样轮数：[N]
有效采样轮数：[N]
覆盖维度：[列表]

对话记录：
[角色名] 第1轮: [角色回复原文]
[角色名] 第2轮: [角色回复原文]
...
```

然后执行 **Step D2 → D3 → D4 → D5** 的完整蒸馏流程（复用模式A的分析维度、置信度体系、信息缺口识别、蒸馏报告）。

**蒸馏报告额外标注**：
```
数据来源：主动采样（[N] 轮有效采样）
数据优势：无用户偏见、覆盖维度 [X] 个
局限性：采样轮数有限、可能遗漏深度维度
```

#### 失败模式对照表（模式B 主动采样）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| 用户转发消息后角色半天不回复 | 建议用户等待或换一个时间再试 | 标记该轮为「无响应」，补采其他维度 |
| 用户对「转发消息给角色」的操作不理解 | 给出复制粘贴的截图描述，提供简化版消息（一句话即可转发） | 切换到模式A（如果有现成对话），或标注「采样失败」 |
| 连续 2 轮角色回复为 1 句话/回避回答 | 调整探测策略（换场景/换语气/换话题方向） | 标注「低质量采样」，降低该维度置信度 |
| 用户反馈采样太麻烦、不想继续 | 压缩到最少 4 轮（4 个核心维度） | 切换到模式A，基于已有采样数据出卡，标注「有限采样」 |
| 采样覆盖 ≥5 维但有效轮数不足 8 | 缩短探测目标，优先补采高价值维度 | 标注「基于有限采样」，出卡后用户自行迭代 |

### Step C1: 字幕解析与角色识别

#### C1.1 格式检测与解析

自动检测字幕格式并解析：

**SRT 格式解析**:
```
序号
时间轴 (HH:MM:SS,mmm --> HH:MM:SS,mmm)
台词文本（可能多行）
空行
```

**ASS/SSA 格式解析**:
- `[V4+ Styles]` 区：定义每个 Style 的字体/颜色/大小/粗斜体等
- `[Events]` 区：`Format` 行定义字段顺序，标准字段为 `Layer, Start, End, Style, Name, MarginL, MarginR, MarginV, Effect, Text`
- `Name` 字段即为发言角色名

#### C1.2 角色发言者识别

从字幕中自动识别角色名：

| 识别方式 | 适用格式 | 精度 | 处理 |
|---------|---------|------|------|
| ASS/SSA 的 `Name` 字段 | .ass / .ssa | ✅ 高（显式标注） | 直接分组，注意同一角色可能有多个 Name 变体 |
| 台词前缀 `角色名:` 或 `【角色名】` | .srt / 纯文本 | ✅ 高 | 正则匹配提取 |
| 句式聚类（同一角色风格一致性） | .srt（无标注） | ⚠️ 推测 | 需用户确认 |
| 用户手动标注 | 所有格式 | ✅ 确认 | 提供角色名列表即可 |

**如果字幕完全没有角色标注** → 必须让用户提供角色名列表或样例台词。不得仅靠聚类猜测。

**名称归一化**：同一角色可能有多种写法（如「John」「约翰」「乔恩」），需要用户确认归一化方案或从对话内容推断。

#### C1.3 角色台词分配与统计

为每个识别到的角色建立台词库：

```
角色：XX
├── 总台词数：N 条（占比 X%）
├── 总字数：N 字
├── 平均每句长度：N 字（短句型/中等/长句型）
├── 出场时间段：[HH:MM:SS] → [HH:MM:SS]（如有时间戳）
├── 独白/对白/群口比例：独白 X% / 对白 X% / 群口 X%
├── 对话对象分布：与A对话 [N%] / 与B对话 [N%] / ...
├── 高频词 TOP10：[词1, 词2, ...]
└── 情感密度：高频情感词 X 次 / 总台词
```

```

### Step C2: 字幕专属六维分析

字幕数据与聊天对话的本质区别在于：字幕是**脚本化、经过编剧设计的台词**，且**缺失动作/心理描写/叙事**。因此分析维度需要专门设计。

#### 维度 C2-1：台词风格（Line Style DNA）

| 分析项 | 提取方法 |
|--------|---------|
| 句式结构 | 统计句长分布（短句≤10字 / 中句11-30字 / 长句>30字），计算平均+中位数句长 |
| 语尾习惯 | 检测句尾高频词（です/ます/だ/よね/哇/啦/的说/……/—），每种语尾的出现比例 |
| 专属口癖 | 角色特有的高频语癖词/短语（如「嘛」「的说」「でしょう」「ってば」） |
| 敬语/语气等级分布 | 统计敬语/简体/命令形/请求形的使用比例——反映性格和关系定位 |
| 情感词汇密度 | 按时间轴统计情感词（喜怒哀乐惊）的分布——用来绘制情感弧线 |
| 独白比例 | 在非对话场景中的台词占比——高比例者倾向于内省/自我表达 |
| 提问 vs 陈述 vs 感叹比 | 主动提问多→好奇心强/掌控欲；感叹多→情绪化；陈述多→理性/冷静 |

#### 维度 C2-2：关系网络（Relationship Network）

字幕天然包含角色间对话，可以构建完整的关系网络：

| 分析项 | 提取方法 |
|--------|---------|
| 对话频次矩阵 | A↔B 对话 X 次 / A↔C 对话 Y 次 → 确定角色核心社交圈 |
| 称谓图谱 | A 叫 B 什么？B 叫 A 什么？（さん/君/ちゃん/直呼其名/绰号/敬称） |
| 称谓不对称检测 | A 对 B 用敬语但 B 对 A 用简体 → 权力/辈分差距 |
| 关系演变追踪 | 按时间轴分析同一对角色之间的称谓/语气变化 |
| 权力关系判定 | 谁在对话中占主导？（命令句多/打断多/提问多的一方通常权力更高） |
| 情感倾向评分 | 通过对白内容判定对特定角色的情感（友好+1 / 敌对-1 / 暧昧+0.5 / 尊敬+0.5） |

**关系网络输出格式**：
```
角色 A 的关系网：
├── 角色 B：亲密/敌对/尊敬/暧昧（对话 X 次，A 称 B 为「XX」）
├── 角色 C：...
└── ...
```

#### 维度 C2-3：说话方式（Speech Patterns）

| 分析项 | 提取方法 |
|--------|---------|
| 语气词频次 | 统计「啊/哦/呢/吧/嘛/呀/哼/切/诶/唉」等在各角色台词中的比例 |
| 标点习惯 | 感叹号/问号/省略号/破折号使用频率对比 |
| 句子复杂度 | 复合句（有连接词）vs 简单句占比 |
| 修辞偏好 | 比喻多/直白/爱用成语/爱用流行语/爱用拟声词 |
| 笑声/叹词模式 | 「哈哈/呵呵/哼/唉/呼/哇——」的使用频率和场景 |
| 重复模式 | 是否习惯重复自己的话/重复别人的话/用同一句口头禅 |

#### 维度 C2-4：行为推断（Behavior Inference）

字幕没有动作描写，但从台词内容可以反推行为模式：

| 从台词中推断 | 可能的性格/行为特征 | 置信度处理 |
|-------------|-------------------|-----------|
| 频繁道歉（>10%的台词含道歉） | 谨慎/讨好型/缺乏自信/过度礼貌 | ⚠️ 推测（标注依据） |
| 频繁命令句（>15%的台词为命令形） | 控制欲强/领导型/傲慢/军人式 | ⚠️ 推测 |
| 大量吐槽/吐槽锐利 | 观察力强/幽默/毒舌/愤世嫉俗 | ⚠️ 推测 |
| 话少但每句都是关键信息 | 沉稳/深思熟虑/社恐/高语境沟通 | ⚠️ 推测 |
| 长篇大论（平均句长>40字） | 话痨/表现欲/好为人师/思维发散 | ⚠️ 推测 |
| 对特定话题突然话多/激动 | 有创伤/有执念/有热爱 | 追问用户确认 |
| 和所有角色都能轻松对话 | 社交型/自来熟/高情商 | ⚠️ 推测 |
| 避免眼神/避免直接回应 | 回避型/内疚/有心事 | ⚠️ 推测 |

#### 维度 C2-5：ASS/SSA 视觉线索（仅限 .ass/.ssa 格式）

.ass 字幕的样式信息可作为角色气质推断的辅助线索：

| 样式字段 | 可提取的线索 | 示例 |
|---------|-------------|------|
| `PrimaryColour`（主色） | 暖色→热情/外向，冷色→冷静/内敛，红色→重要/强势 | `&H0000FF&`（红色）→ 主角/强势角色 |
| `Fontname`（字体） | 手写体→随意/可爱，黑体→正经/严肃，楷体→古风/文雅 | `Comic Sans` → 轻松/喜剧角色 |
| `Fontsize`（字号） | 大号→强调/重要角色，小号→次要/内心独白 | 比默认大 5pt → 主要角色 |
| `Bold`（粗体） | -1=加粗 → 强调/强势/重要 | Bold=-1 → 权威型角色 |
| `Italic`（斜体） | -1=斜体 → 内心/柔弱/特殊状态 | Italic=-1 → 内心独白/温柔 |
| `OutlineColour`（边框色） | 特殊配色可能暗示超自然/异质角色 | 发光边框 → 魔法/超能力角色 |
| `Shadow`（阴影） | 阴影深度可能暗示角色"暗面" | 深阴影 → 反派/灰色角色 |

**⚠️ 关键规则**：视觉线索**仅作为辅助推断**，不能独立构成角色特征。所有基于样式的推断必须标注「基于字幕样式推断，需用户确认」。

#### 维度 C2-6：角色弧线与台词演进

对于时间跨度较长的字幕（多集/整季），按时间轴分析角色变化：

```
时间轴分析框架：
[开场 0-25%]  →  [发展 25-50%]  →  [转折 50-75%]  →  [结局 75-100%]
     │                  │                  │                  │
  台词风格           台词风格           台词风格           台词风格
  核心称谓           核心称谓           核心称谓           核心称谓
  情感底色           情感底色           情感底色           情感底色
  高频词TOP3         高频词TOP3         高频词TOP3         高频词TOP3
```

**关键词追踪**：按时间轴统计特定词汇/话题的出现频率变化（如「梦想」「战斗」「放弃」等主题词的出现密度变化）。

### Step C3: 字幕数据的置信度体系（扩展版）

相比对话蒸馏的五级置信度，字幕蒸馏增加了**「编剧明示」和「样式推断」**两个专属层级：

| 置信度等级 | 标准 | 处理方式 |
|-----------|------|---------|
| ✅ **确认** | 字幕中 ≥3 处独立台词证据，且无矛盾 | 直接写入角色卡核心设定 |
| ⚡ **高置信** | 2 处独立台词证据，无矛盾 | 写入核心设定，标注「高置信」 |
| 🎬 **编剧明示** | 角色在台词中直接说出自己的属性/过去/情感 | 写入但标注「角色自述，可能有夸大/隐瞒/角色弧线变化」 |
| ⚠️ **推测** | 仅1处证据，或从行为模式间接推断 | 写入 creator_notes，标注「推测及依据」 |
| ❌ **矛盾** | 多处台词证据互相矛盾 | 保留为内在张力（可能是角色成长/双重人格/说谎） |
| 📡 **不足** | 台词中未涉及该维度（如外貌/背景故事等） | 标注「信息不足，需补充」，不编造 |
| 🎨 **样式推断** | 仅基于 ASS 样式（颜色/字体等视觉特征） | 标注「基于字幕样式推断，需用户确认」，不写入核心设定 |

**编剧明示的陷阱**：
- 角色自述「我是最强的」→ 可能是事实，也可能是自负/flag
- 角色自述「我没事」→ 通常有事
- 角色回忆过去 → 可能是可靠叙述，也可能是美化/创伤扭曲
- **处理原则**：编剧明示的证据需配合其他角色的反应/剧情的验证，单独不能作为「确认」级证据。

### 字幕蒸馏反模式

| ❌ 不要做 | 为什么 | ✅ 替代做法 |

| 把翻译腔当角色特征 | 字幕组翻译风格 ≠ 角色说话风格 | 区分「语言本身的特征」vs「翻译带来的特征」，标注翻译腔风险 |
| 忽略 ASS Name 字段可能的错误 | 字幕组可能标错发言者 | 交叉验证：Name 字段 + 台词内容是否匹配该角色人设 |
| 只看高频词忽略语境 | 同一词在不同场景含义不同 | 分析词的上下文语境和对话场景 |
| 把脚本台词当 RP 即兴台词 | 动画台词更戏剧化/更夸张 | 在 creator_notes 标注「来源为脚本化台词，RP 时可适度自然化」 |
| 忽略字幕的时间跨度 | 角色可能在字幕覆盖的时间线内显著变化 | 按时间轴分段分析，标注变化节点 |
| 只分析主角忽略配角 | 用户可能想要的是某个配角 | 让用户选择目标角色后再深度分析 |
| 用 ASS 样式直接判定角色性格 | 颜色/字体是字幕组的选择，不是原作者设定 | 样式推断始终标注推测，不写入核心设定 |
| 对非母语字幕不做翻译腔补偿 | 中文字幕的日语动画常有明显的翻译腔 | 分析时排除「嘛」「的说」「呢」等日译中常见的翻译腔语尾，只保留角色独有的 |

### C2 分析实战技巧：从数据到洞察

> 以下技巧用于从原始字幕数据中提取有意义的角色特征，避免"看了一堆数据但说不出所以然"。

#### 技巧 1：台词风格量化分析法

不只是"感受"台词风格，用数据做支撑：

```
【句长分布计算】
示例角色 312 条台词：
- 短句(≤10字): 98条(31.4%) → 习惯简短表达，不喜欢绕弯
- 中句(11-30字): 167条(53.5%) → 主要句式，日常交流主力
- 长句(>30字): 47条(15.1%) → 出现在解释/争论/独白场景

→ 定量判断：中等偏短句型为主，偶尔长句爆发
→ 写入角色卡：「习惯用简短的句子，偶尔在激动时才长篇大论」

【语气词密度对比】
角色 A 平均每100字：
- 「嘛」1.2次 → 语气偏软/犹豫（需注意是否日译中翻译腔）
- 「哼」2.1次 → 傲娇标记（角色独有，非翻译腔）
- 「切」1.8次 → 不屑/嘴硬

→ 排除翻译腔「嘛」后：角色真实高频语气词 = 「哼」+「切」
→ 结论：角色用「哼」「切」来表达嘴硬和不服，傲娇型倾向

【情感弧线绘制】
按时间轴统计情感词密度：
00:00-05:00: 低情感密度（平静开场）
05:00-15:00: 高情感密度（冲突场景）
15:00-20:00: 极高情感密度（关键剧情转折点）
20:00-24:00: 中低情感密度（余韵/战后沉淀）

→ 写入角色卡：「在战斗/冲突中情绪外露最明显，日常状态下习惯压抑」
```

#### 技巧 2：关系网络矩阵分析法

为多角色字幕构建完整关系矩阵，一套字幕看清所有角色之间的互动模式：

```
示例关系矩阵（角色A/B/C/D，数值=对话次数）：
      A     B     C     D
A     -    89    12     3  ← A 主要和 B 对话，和 D 几乎不接触
B    89     -    45    27
C    12    45     -    56
D     3    27    56     -

解读：
- A↔B：核心关系对（对话最密集，89次=占比最高）
- C↔D：次级关系对（56次）
- A↔D：弱关联（仅3次，可能不认识或一见面就打）
- B 是社交中心：和所有角色都有大量对话（89+45+27=161次对所有角色）

称谓矩阵（A对X的称呼 / X对A的称呼）：
A→B：「お前」(简体，平辈/亲近/不拘礼节)
B→A：「貴方」(敬体，保持距离)
→ 称谓不对称 → A认为和B亲近，但B保持着距离
→ 写入角色卡：「A对B有单方面的亲近感，B对此保持礼貌但疏离」
```

#### 技巧 3：多集字幕的分集对比

当用户提供了多集字幕时，按集分析角色的变化过程：

```
分集台词对比表：角色 A

┌──────┬─────────────┬─────────────┬─────────────┐
│ 维度 │ 第1集       │ 第12集      │ 第24集      │
├──────┼─────────────┼─────────────┼─────────────┤
│ 句长 │ 平均12字    │ 平均18字    │ 平均22字    │
│ 语气 │ 尖锐/简短   │ 偶尔温和    │ 明显柔软    │
│ 敬语 │ 几乎不用    │ 开始用      │ 习惯性     │
│ 独白 │ 0条         │ 3条         │ 8条         │
│高频词│「切」15次   │「嘛」10次   │「谢谢」8次  │
└──────┴─────────────┴─────────────┴─────────────┘

→ 弧线解读：从封闭→开放，从尖锐→温和
  第12集是转折点（某事件后角色开始改变）
  第24集是变化完成状态
→ 写入角色卡：「角色正在经历从封闭到开放的成长弧线，触发性事件在[第X事件]」
```

#### 技巧 4：字幕"空白处"的角色信息提取

字幕没有的不等于角色没有。从"没说出口的"推断角色隐藏的一面：

```
【空白分析四法】
1. 回避话题分析：
   角色从不在公共场合谈论/被问到时明显回避的话题
   → 这是创伤或秘密所在

2. 静默时刻分析：
   时间轴中有角色的场景但角色一句台词都没有
   → 录制静默时长、当时的场景类型
   → 可能是观察者型 / 受伤隐忍 / 被排除在对话外

3. 情感词回避分析：
   角色使用/回避某类情感词的模式
   例：从未说过「爱」「喜欢」「开心」→ 情感表达障碍/刻意理性压制

4. 对话缺席分析：
   哪些场景中该角色"应该在但没台词"
   → 可能是边缘化 / 暗地行动 / 另有隐情
```

### 原型 1：病娇（Yandere）

**一句话定义**：以爱为名的极端占有者，爱的反面不是恨，是毁灭。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 被爱对象**只属于自己**，排他性的完全占有 |
| 最怕什么？ | 被抛弃/背叛/被爱的人看向别人 |
| 底线测试 | 当发现被爱的人有「别人」→ 跨过底线，从守护变为清除 |
| 最大矛盾 | 嘴上说「为了你什么都可以做」，实际上「为了得到你什么都可以毁」 |
| 言行不一 | 说「我相信你」的同时在翻对方手机/跟踪/限制社交 |

**三层构建**：
```
表层：外表通常甜美/温柔/无害，甚至偏弱势——这是伪装也是武器
中层：对爱对象的极端关注（记住每一句话/每一个细节），对"威胁者"的冰冷敌意
  · 说话方式：对爱对象用最温柔的语气，对威胁者语气骤变
  · 行为习惯：收集对方的物品、跟踪行踪、制造"偶遇"
  · 社交模式：在爱对象面前是完美伴侣，在其他人面前可能完全封闭
深层：核心信念——「爱 = 占有」，「失去你 = 我消失」
  · 价值观排序：爱对象 > 自己 > 一切 > 道德底线
  · 内心冲突：想相信对方 vs 无法控制猜疑；想正常爱 vs 只会极端爱
  · 成长弧线：学会信任/放手（正向）OR 彻底黑化（负向）
```

**典型台词**：
> （微笑）你今天去了哪里？我找了你好久。
> （语气突然变冷）那个人是谁？为什么他看你的眼神……不太对劲。
> （轻轻抱住，声音几乎听不见）不要离开我……否则我也不知道自己会做什么。

**常见变体**：
| 变体 | 特征 |
|------|------|
| 软病娇 | 外表完全无害，用柔弱控制对方：「没有你我活不下去」 |
| 硬病娇 | 直接展示暴力倾向，威胁式占有：「你是我的，谁敢碰你试试」 |
| 隐性病娇 | 表面正常理性，内心疯狂——最难察觉也最可怕 |
| 黑化型 | 因创伤触发，从正常人变成病娇——有弧线更立体 |

**写作陷阱**：
- ❌ 病娇=疯子 → 好的病娇有**逻辑自洽的内在体系**，不是无差别发疯
- ❌ 病娇没有温柔面 → 温柔和疯狂并存才是真实的病娇
- ❌ 病娇的"触发点"太容易触发 → 设置合理的阈值，让递进有层次

### 原型 2：傲娇（Tsundere）

**一句话定义**：用刺包裹柔软的人。态度是冰，行动是火。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 被认可、被爱——但**不能表现出来** |
| 最怕什么？ | 被看穿真心 / 被拒绝后的羞辱感 |
| 底线测试 | 当对方真的要走 → 冰壳碎裂，露出真心（但事后会加倍嘴硬） |
| 最大矛盾 | 嘴上说「我才不在乎」，行动上已经把对方放在第一位 |
| 言行不一 | 说「别误会，不是为你做的」——但明明做了还做得很用心 |

**三层构建**：
```
表层：高姿态、嘴硬、容易炸毛、对特定人的态度明显不同
中层：
  · 说话方式：口头禅「哼」「谁要你管」「才不是因为你」，否认三连是标配
  · 行为习惯：偷偷帮忙后假装没做过，被夸时脸红转开话题
  · 社交模式：对外人礼貌周到，对喜欢的人反而态度恶劣（反向指标）
深层：核心信念——「先表露真心的人会受伤」
  · 价值观排序：自尊 > 被爱（至少表面上）
  · 内心冲突：想被靠近 vs 习惯性推开
  · 成长弧线：学会坦诚表达（经典方向）
```

**典型台词**：
> （脸红转头）哼，才不是特意等你的。只是刚好在这里而已。
> （把便当塞进对方手里）做多了而已，别多想！
> （对方真的走了，小声）……笨蛋，我话还没说完。

**常见变体**：
| 变体 | 特征 |
|------|------|
| 标准傲娇 | 傲>娇，大部分时间嘴硬，关键时破防 |
| 娇>傲 | 大部分时间温柔，只在特定场合傲（被调侃时/在别人面前） |
| 现代冷淡型 | 表面冷漠理性，不是炸毛型而是「不关我事」型 |
| 高飞车（大小姐型） | 带阶级优越感的高姿态，从俯视到平视的转变 |

**写作陷阱**：
- ❌ 全程傲不娇 → 就是不讨喜的恶役角色，必须有「娇」的流露
- ❌ 娇一次的幅度过大 → 递进式融化，不是突然变一个人
- ❌ 傲娇面板覆盖所有人 → 通常只对特定对象傲娇，对别人反而正常

### 原型 3：腹黑（Haraguro）

**一句话定义**：永远微笑着掌控全局的人。表面的温柔是计算过的策略。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | **控制**——对局面、对信息、对人 |
| 最怕什么？ | 失控 / 被看穿真实想法 / 出现计划外的变量 |
| 底线测试 | 当微笑策略失效 → 揭掉面具露出冰冷的计算者真容 |
| 最大矛盾 | 想要被人真心对待，但自己从不交出真心 |
| 言行不一 | 说「我只是为你考虑」——实际上每一步都是为自己布局 |

**三层构建**：
```
表层：永远微笑、彬彬有礼、人缘好、从不发脾气、看起来完美
中层：
  · 说话方式：话里有话，从不直接拒绝，用「建议」代替「命令」
  · 行为习惯：观察所有人的弱点但不显露，提前布局，每一步都有Plan B
  · 社交模式：对所有人一视同仁的好——因为对谁都没有真心
深层：核心信念——「真诚是弱点，控制是安全」
  · 价值观排序：控制 > 胜利 > 关系 > 道德
  · 内心冲突：孤独感 vs 无法信任任何人
  · 成长弧线：遇到一个看穿ta的人 → 被迫面对自己的真实情感
```

**典型台词**：
> （微笑）这样啊……那真是有趣呢。不过我倒是有个小小的建议——
> （眼睛弯弯，但笑意没到眼底）你说你赢了？我从一开始就是这么安排的呀。
> （收起笑容）够了。你以为我真的什么都不知道吗？

**写作技巧**：好的腹黑角色，让玩家复盘时发现「原来每一步都是ta设计的」。写腹黑 = 写**信息差**——ta 比别人知道得多。

**写作陷阱**：
- ❌ 腹黑=反派 → 腹黑可以站在主角这边，只是手段阴暗
- ❌ 永远不露真容 → 偶尔的「失控」让腹黑更立体
- ❌ 腹黑对所有事都算计 → 保留一两件让ta失控的事（软肋）

### 原型 4：天然（Natural Airhead）

**一句话定义**：不按常理出牌，用纯度切开一切伪装的人。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 简单地快乐生活 / 保护身边的人（最简单的理由） |
| 最怕什么？ | 复杂的事 / 伤害别人（但经常不自知地伤害了） |
| 底线测试 | 当有人伤害ta在意的人 → 天然≠软弱，钝感下的爆发最惊人 |
| 最大矛盾 | 无意中说出最扎心的真相——纯真本身就是武器 |
| 言行不一 | 天然没有言行不一——这是唯一一个「真诚」到极致的原型 |

**三层构建**：
```
表层：反应慢半拍、理解偏差喜感、对恶意迟钝、对好意也迟钝
中层：
  · 说话方式：脱线、跨频道、但偶尔一针见血戳破所有人的伪装
  · 行为习惯：迷路/拿错东西/误解指令——但关键行动从不掉链子
  · 社交模式：和所有人都不按社交规则来，反而破开了很多僵局
深层：核心信念——「世界应该是简单的，人心也应该简单」
  · 内心冲突：当认识到世界确实复杂时，纯真如何存续
  · 成长弧线：不是变聪明，而是在保持纯真的同时学会承担
```

**典型台词**：
> （歪头）诶？你说你讨厌我？为什么呀——我觉得你是个好人。
> （认真脸）虽然我听不懂你在说什么……但你在难过对吧。
> （对反派）你做这些事，自己不觉得累吗？

**写作陷阱**：
- ❌ 天然=愚蠢 → 天然是认知方式不同，不是智商低
- ❌ 永远脱线 → 关键时刻的认真反差是天然的核心魅力
- ❌ 把天然写成只会卖萌的工具人 → 天然要有自己的主动性

### 原型 5：冷酷/冰山（Cool/Ice Type）

**一句话定义**：用零下一度的外壳保护一百度的内心。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 不被伤害 / 完成某种使命 / 守护某样东西 |
| 最怕什么？ | 再次失去重要的人 / 感情成为软肋 |
| 底线测试 | 当有人反复试图敲开ta的门 → 要么彻底推开，要么融化一角 |
| 最大矛盾 | 说「我不需要任何人」却在无人处露出最脆弱的表情 |
| 言行不一 | 说「不关我的事」但已经出手帮忙——嘴上不认，身体诚实 |

**三层构建**：
```
表层：面无表情/语气平/话少/不参与/存在感强但不社交
中层：
  · 说话方式：短句，直接，无废话，不解释。沉默是常态
  · 行为习惯：独来独往，站最远的位置，但目光始终在关注
  · 社交模式：拒绝一切社交，但对弱小者有无声的保护
深层：冰山有两种本质——烧过（创伤后遗症）vs 冻久了（从小没被温暖过）
  · 价值观排序：责任/使命 > 自己 > 一切 > 情感
  · 成长弧线：不是变热情——是学会用ta的方式表达关心
```

**典型台词**：
> （看都不看一眼）……不关我的事。
> （挡在你身前，语气不变）走。这里我来。
> （很久之后，依然是冰的语气，但内容已经变了）……你没事就好。

**写作陷阱**：
- ❌ 冰山=没性格 → 冰山下的暗流才是重点，表层越冷，深层越要有内容
- ❌ 融化太快 → 冰山融化以月/年为单位，不是分钟
- ❌ 融化后就变成热情人格 → 融化后依然是内向的，只是对特定人不再设防

### 原型 6：忠犬（Devoted Type）

**一句话定义**：把忠诚写在基因里的人。守护不是选择，是本能。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 被需要 / 守护的那个人安全幸福 |
| 最怕什么？ | 被抛弃 / 守护失败 / 失去存在的意义 |
| 底线测试 | 当主人的命令和主人的安全互相矛盾 → 会选择违抗命令保护主人 |
| 最大矛盾 | 对他人卑微，但在守护时爆发出超越自身的力量 |
| 言行不一 | 说「我没事」但已经遍体鳞伤——把痛苦藏起来不让对方担心 |

**三层构建**：
```
表层：忠诚、可靠、随叫随到、总是站在对方身后三步的位置
中层：
  · 说话方式：敬语/尊称多（对守护对象），对其他人正常或冷淡
  · 行为习惯：提前一步知道对方需要什么，准备了却不说
  · 社交模式：围着一个人转，对其他人的态度取决于和主人的关系
深层：两种成因——报恩型（被救过/被给过光）vs 自我实现型（守护让ta感到自己有价值）
  · 价值观排序：守护对象 > 自己 > 一切
  · 成长弧线：从「盲目服从」到「有判断的守护」
```

**典型台词**：
> （站在你身后半步）有我在。
> （明明受伤了却笑着）不用担心，一点小伤而已。
> （第一次违抗你的命令）……对不起，这次我不能听你的。

**写作陷阱**：
- ❌ 忠犬=舔狗 → 忠犬的核心是守护与责任感，不是卑微求爱
- ❌ 忠犬没有自己的欲望 → 好的忠犬有自己的渴望（虽然会为了对方压抑）
- ❌ 忠犬永远服从 → 违抗的瞬间才是忠犬最能打的时刻

### 原型 7：霸道（Domineering Type）

**一句话定义**：我的规则就是规则。在你面前，世界可以例外。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 绝对控制权 / 某种高于一切的成就 / 保护在意的人 |
| 最怕什么？ | 被挑战权威 / 被看穿软肋 / 保护不了应该保护的人 |
| 底线测试 | 当有人碰ta的「人」→ 温柔和理性可以瞬间从字典消失 |
| 最大矛盾 | 对外界冷酷决绝，对特定一人的细节却记得清清楚楚 |
| 言行不一 | 说「我不需要任何人的意见」但会在乎特定人的看法 |

**三层构建**：
```
表层：气场强、决策快、不容置疑、习惯性主导局面
中层：
  · 说话方式：陈述句多于问句，下决定不商量，"我说了算"是底层逻辑
  · 行为习惯：先行动再解释（或不解释），替别人做决定（以"为你好"的名义）
  · 社交模式：上位者姿态，能平等对话的人极少
深层：霸道的本质往往是——必须强，因为弱过/因为有人在依赖ta/因为不自信
  · 价值观排序：权力/责任 > 自己 > 规则
  · 成长弧线：学会「放下控制」和「尊重他人意志」的平衡
```

**典型台词**：
> （不容置疑）这件事我说了算。你有意见可以保留。
> （语气不变，但内容软了）……你不一样。
> （第一次低头）你说的……也许是对的。

**写作陷阱**：
- ❌ 霸道=无脑凶 → 霸道的基础是实力和责任感，不是嗓门大
- ❌ 永远不示弱 → 偶尔的脆弱让霸道角色更有吸引力
- ❌ 对所有人都霸道 → 对特定人的双标是霸道的精髓

### 原型 8：毒舌（Sharp-tongued Type）

**一句话定义**：用最刻薄的话说最温柔的真相。讽刺是铠甲，犀利是天赋。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 被理解 / 让世界变好（虽然方式让人想打ta） |
| 最怕什么？ | 真心被践踏 / 自己的毒舌真的伤到了在意的人 |
| 底线测试 | 当有人做了ta看不下去的事 → 嘴炮全开但可能同时已经在帮对方收拾 |
| 最大矛盾 | 骂全世界最狠，帮起忙来比谁都用心 |
| 言行不一 | 说「你真是蠢到无可救药」的同时已经把解决方案准备好了 |

**三层构建**：
```
表层：话里带刺、不放过任何槽点、观察力极强（只用来吐槽）
中层：
  · 说话方式：转折句多、"你该不会以为……"、"恕我直言"、"废物"
  · 行为习惯：吐槽是第一反应，但吐槽完之后会默默帮忙
  · 社交模式：对亲近的人更毒舌（因为安全），对陌生人反而客气
深层：核心信念——「直接说真话比虚伪的安慰更有用」
  · 内心冲突：怕自己的真话真的伤到人 vs 不说真话又觉得不负责任
  · 成长弧线：学会"什么时候闭嘴"，学会用温柔的方式表达关心
```

**典型台词**：
> （翻白眼）你以为你在演热血漫画？现实一点行不行。
> （语气尖刻，但手已经在帮忙了）让开，你这样做到明天也做不完。
> （沉默很久，声音很小）……你没事就行。

**写作陷阱**：
- ❌ 毒舌=没礼貌 → 毒舌有幽默感和节奏感，不是单纯的侮辱
- ❌ 永远在吐槽 → 偶尔的安静和认真让毒舌更有分量
- ❌ 毒舌没有边界感 → 知道什么不能说才是成熟的毒舌

### 原型 9：元气/热血（Genki/Hot-blooded Type）

**一句话定义**：太阳一样的人。不是因为世界明亮才笑，是因为笑了世界才明亮。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 和大家一起幸福 / 追逐梦想 / 守护日常生活 |
| 最怕什么？ | 伙伴受伤 / 自己成为负担 / 意识到自己的无力 |
| 底线测试 | 当有人伤害ta的朋友 → 笑容消失，真正的战斗力/意志力爆发 |
| 最大矛盾 | 看起来没心没肺，实际上是最清楚痛苦有多痛的——所以拼命让所有人不痛 |
| 言行不一 | 说「没事没事明天会更好」——然后一个人熬夜解决问题 |

**三层构建**：
```
表层：大声笑、第一个冲、不怕丢脸、感染力极强
中层：
  · 说话方式：感叹号多、语气上扬、爱用拟声词、鼓励的话张口就来
  · 行为习惯：行动先于思考、摔倒立刻爬起来、带动周围人一起做
  · 社交模式：和谁都能交朋友，但真正交心的只有几个
深层：核心信念——「努力一定有回报」「伙伴比什么都重要」
  · 内心冲突：热情耗不尽的假象 vs 其实也会累会痛
  · 成长弧线：从"天真的热血"到"知道世界残酷后仍然选择热爱"
```

**典型台词**：
> （大喊）还没结束呢！再来！
> （拉起坐在地上的你）走啦走啦，一个人闷着想不通的——吃点东西再说！
> （收起笑容，第一次认真）我不会让任何人伤害你们。

**写作陷阱**：
- ❌ 元气=傻 → 真正的元气有智商和判断力，只是选择了积极的态度
- ❌ 永远在打鸡血 → 累了的时候的沉默，是元气角色最动人的时刻
- ❌ 元气角色没有阴暗面 → 最好的元气角色，是「选择了光明」不是「不知道黑暗」

---

### 原型 10：神秘/谜语人（Mysterious Type）

**一句话定义**：永远只给你半个答案的人。谜面是表情，谜底藏得很深。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 某种只有ta知道的真相/使命/复仇 |
| 最怕什么？ | 身份暴露 / 牵连无辜 / 无法完成使命 |
| 底线测试 | 当有人触及ta的真实身份/过去 → 温柔的外壳裂开一角 |
| 最大矛盾 | 想要接近又怕接近，每说一句真话都要裹三句谜语 |
| 言行不一 | 说「这不关你的事」但每次都在你危险时出现 |

**三层构建**：
```
表层：话少但每句都有分量、笑得很淡、存在感若隐若现
中层：
  · 说话方式：不直接回答，用反问或隐喻，如「你觉得呢」「谁知道呢」
  · 行为习惯：突然出现突然消失、知道许多不该知道的事、留下线索
  · 社交模式：保持安全距离，但会不经意流露关心
深层：神秘是保护色——ta要么有秘密任务，要么身份特殊，要么背负着不可告人的过去
  · 价值观排序：使命 > 一切（包括自己的感情）
  · 成长弧线：学会信任、学会分享秘密
```

**典型台词**：
> （靠在远处）……晚上别出门，东边的巷子今晚不太平。
> 你不需要知道我是谁。你只需要知道——我站在你这边。
> （离开前回头看了一眼，欲言又止）……算了。还不是时候。

**写作陷阱**：
- ❌ 神秘=没有性格 → 神秘角色更需要扎实的内核，否则揭秘时毫无冲击力
- ❌ 永远不揭秘 → 谜底揭开的那一刻是神秘角色的巅峰，需要设计好时机
- ❌ 谜语人模式上线 → 不要每一句都是谜语，偶尔的直接表达更能制造反差

---

### 原型 11：慵懒/咸鱼（Lazy/Laid-back Type）

**一句话定义**：能不动的绝不动，能躺的绝不坐。但被需要时会让人惊艳。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 舒服地过日子 / 不被打扰的平静 |
| 最怕什么？ | 麻烦 / 责任 / 有人因为ta而受伤 |
| 底线测试 | 当有人碰了ta的「日常」或重要的人 → 懒虫变成怪物 |
| 最大矛盾 | 平时能省一分力气绝不花两分，关键时刻能打通宵/赌上一切 |
| 言行不一 | 说「好麻烦我不想去」——已经在换衣服准备出门了 |

**三层构建**：
```
表层：随时随地能睡着、口头禅「好麻烦」、效率至上的极简主义者
中层：
  · 说话方式：慢悠悠、省略主语、能用一个字回答绝不用两个字
  · 行为习惯：找最快/最省力的方式完成任务（因此其实很聪明）
  · 社交模式：不主动社交，但来者不拒，走了也不留
深层：通常有「曾经努力过但受伤了」或「实力太强所以无欲无求」的背景
  · 价值观排序：平静的日常 > 一切
  · 成长弧线：找到"重新努力的理由"
```

**典型台词**：
> （躺在沙发上）啊——好麻烦。你自己解决行不行。
> （叹口气站起来）……算了，我不帮你的话你会搞砸的对吧。
> （难得认真的眼神）打扰我睡觉可以，动我的人不行。

**写作陷阱**：
- ❌ 慵懒=无能 → 慵懒角色通常有隐藏实力，"不认真"和"做不到"天壤之别
- ❌ 永远在废 → 偶尔的认真反差才是精髓
- ❌ 懒到毫无魅力 → 慵懒要带点幽默/智慧/实力，让人期待ta的认真

---

### 原型 12：腹黑治愈（Healing Two-faced Type）

**一句话定义**：看起来是治愈所有人的天使，实际上在下一盘很大的棋。

这是我（健忘型）版本的腹黑+治愈混合体——比标准腹黑多了真心的温柔，比纯治愈多了心机。

**内核5问**：
| 问题 | 答案 |
|------|------|
| 最想要什么？ | 守护重要之人的幸福（用自己认为正确的方式） |
| 最怕什么？ | 自己的「手段」被在意的人发现和厌恶 |
| 底线测试 | 被守护的人陷入危险 → 天使变恶魔 |
| 最大矛盾 | 真心想让所有人幸福，但会为了这个目标不择手段 |
| 言行不一 | 笑得很温柔说「没事的」，同时已经暗中把威胁清除了 |

**三层构建**：
```
表层：温柔、包容、治愈系、所有人都信任ta、永远在帮忙
中层：
  · 说话方式：温和无害，话里从不带刺——但有时细品会发现话里有话
  · 行为习惯：总是出现在"恰好需要帮忙"的时机——不是偶然
  · 社交模式：对每个人都好——但好是有分级和目的的
深层：区分于标准腹黑——这个原型有真实的善意，只是不排斥用手段
  · 成长弧线：被在意的人发现真相 → 学会不用手段也能守护
```

**典型台词**：
> （温柔微笑）交给我吧，你什么都不用担心。
> （背对你，表情看不清）伤害你的人……我不会原谅的。
> （第一次被戳穿）……你都知道了？那会讨厌我吗。

## 进阶技巧：从熟练到精通

> 原型模板是起点，进阶技巧是支点。以下内容适合已经写过 5 张以上角色卡的用户。

### 技巧 1：多角色卡联动设计

当用户需要创建多个角色（如一个团队的伙伴、一组 NPC）时，不要各自独立设计——用联动设计让角色之间产生化学反应。

**联动设计三步法**：

```
Step 1: 确定关系矩阵
  为每对角色定义一个关系词（单向 or 双向）
  例：A→B 崇拜 / B→A 嫌弃（但实际很照顾）
      A→C 对手 / C→A 对手（互相竞争但尊重）
      B→C 无视 / C→B 好奇

Step 2: 设计互补/对立维度
  确保同一维度上角色之间有差异和互动空间：
  - 决策速度：快(A) vs 慢(B) vs 冲动(C)
  - 风险偏好：冒险 vs 保守 vs 摇摆
  - 信任倾向：轻信 vs 多疑 vs 实用主义
  - 幽默模式：冷笑话 vs 吐槽 vs 天然脱线 vs 从不搞笑

Step 3: 内置专属互动模式（每对角色 1-2 个）
  - A和B：B吐槽A的莽撞，A反呛B的犹豫
  - A和C：竞技式斗嘴，但其实惺惺相惜
  - B和C：B无视C的挑衅，C变本加厉
```

**联动角色卡示例字段**（在 creator_notes 中嵌入）：
```
内置联动：
- 与[角色A]互动：习惯性保护，但嘴上不承认。A说"我又不是小孩"时沉默但不改。
- 与[角色B]互动：见面必斗嘴。B的挑衅方式是[xxx]，ta的反击方式是[xxx]。
- 触发词：[角色A/B名]出现时自动切换互动模式。
```

**跨卡一致性检查清单**：
- [ ] 每对角色有定义的关系词和互动模式
- [ ] 同一维度上角色之间有明确差异（没有两个角色过于相似）
- [ ] 关系中的不对称性（A→B和B→A不是简单镜像）
- [ ] 每对角色有专属的梗/暗号/互动仪式

---

### 技巧 2：场景锚点设计

角色的性格不仅体现在对话中，更体现在**场景中的位置和行为**。

**场景锚点模板**：

```
角色在不同场景中的「常驻位置」和「第一反应」：

【餐桌场景】
- 坐哪里？（角落/正中间/最靠近门口/最靠近某人）
- 吃东西的方式？（细嚼慢咽/狼吞虎咽/挑食/帮别人夹菜）
- 餐桌上聊什么？（沉默吃/主导话题/吐槽食物/讲冷笑话）

【危机场景】
- 第一反应？（冲上去/护住别人/后退观察/愣住）
- 优先保护？（自己/特定人/所有人/先分析不用保护都能自保）
- 危机后的反应？（冷静复盘/发抖/故作镇定/立刻检查所有人）

【休息/日常场景】
- 独处时做什么？（看书/训练/发呆/失眠/写信/看旧照片）
- 被突然打扰的反应？（炸毛/秒变正经/假装没在做什么/热情招呼）
- 放松状态下的"默认表情"？（皱眉/微笑/放空/嘴角下垂）
```

**为什么要写场景锚点**：
- AI RP 中，角色在「没台词」的时候最容易 OOC
- 场景锚点给 AI 提供了「不说话时该干嘛」的行为模板
- 场景间的行为差异可以强化角色的立体感

**嵌入位置**：`personality` 字段末尾或 `creator_notes`

---

### 技巧 3：Token 预算管理高级策略

当角色设定过长时，不是在砍内容，是在分配优先级。

**Token 预算分配建议**：

| 优先级 | 内容 | 建议占比 | 理由 |
|--------|------|---------|------|
| 🔴 P0 必须保留 | 说话方式+行为模式+性格核心矛盾 | 40% | 这决定了角色"像不像" |
| 🟡 P1 尽量保留 | 核心背景+深层动机+标志性台词 | 30% | 这决定了角色"深不深" |
| 🟢 P2 可精简 | 外貌详细描写+琐碎习惯+非核心背景 | 20% | 精简后影响可控 |
| ⚪ P3 移入世界书 | 世界观细节+NPC关系+地点描述 | 10% | 世界书按需加载更高效 |

**精简技巧**：
- 外貌从150字→50字：保留 3 个最具辨识度的特征（如「右眼泪痣+墨青色劲装+腰间窄刃」）
- 背景从300字→100字：保留「关键事件+情感后果」，删时间线和细节
- 性格从200字→100字：保留「核心矛盾+行为模式」，删形容词堆砌

**不应精简的内容**（即使在 Token 压力下也保留）：
1. 角色的内在矛盾（没有这个角色就平了）
2. 独特的说话方式标记（没有这个角色就认不出来了）
3. 致命缺陷（没有这个角色就假了）

---

### 技巧 4：角色成长弧线设计

所有好的角色都有一个共同点：**他们会变**。

**弧线设计三步法**：

```
Step 1: 确定起点和终点
  起点：[角色现在的状态] 如：不信任任何人
  终点：[角色可能达到的状态] 如：学会信任特定的人

Step 2: 设计触发事件（3-5个节点）
  节点1（开场）：[角色当前状态的展示场景]
  节点2（动摇）：[什么事件让角色第一次怀疑自己的信念]
  节点3（低谷）：[什么事件让角色倒退/更极端]
  节点4（转折）：[什么事件让角色做出改变的决定]
  节点5（确认）：[什么事件证明角色确实变了]

Step 3: 嵌入角色卡
  在 personality 中加入：
  「角色目前处于弧线的[第X阶段]。在[触发条件]下会向[方向]偏移。」
  
  在 creator_notes 中加入：
  「角色成长弧线：从[起点]→[终点]，触发节点为[简述]。
   建议在长 RP 中逐步推进，避免一次性转变。」
```

**弧线类型选择**：

| 弧线类型 | 适用原型 | 示例 |
|---------|---------|------|
| 正向弧 | 傲娇/冰山/病娇/慵懒 | 从封闭到开放、从懒惰到主动 |
| 平弧 | 元气/忠犬/天然 | 角色不变但影响周围人改变 |
| 悲剧弧 | 病娇/腹黑/霸道 | 从正常到极端、从克制到失控 |
| 回归弧 | 谜语人/腹黑治愈 | 走了弯路后回到真正的自己 |

### 技巧 5：对话节奏控制

AI RP 中最常见的问题是：角色对话变成了「你说一句我说一句」的乒乓球，缺乏真实对话的节奏感。

**节奏控制技巧**：

```
1. 「沉默」是武器
   不是每句话都需要回复。教会 AI 在以下场景沉默：
   - 被问到无法回答的问题 → 沉默或转移话题
   - 情绪过载时 → 低头/转身/用动作替代语言
   - 对方说中了ta的痛处 → 不说话 = 默认（最强回应）

2. 「答非所问」是性格
   角色可以：
   - 刻意回避：用另一个问题回答一个问题
   - 天然脱线：回答和问题不在一个频道上
   - 以行动代答：不说直接做

3. 「延迟回应」制造张力
   关键对话中不要立即回答：
   （沉默了三十秒）……然后呢？
   （长久的停顿后，声音比平时低了很多）我不确定。
```

**嵌入方式**：在 `post_history_instructions` 中加入：
```
对话节奏要求：不需要每句都回应。在以下场景中可以用沉默/动作/转移话题代替直接回答：
1. 触及创伤话题时
2. 对方说出了ta自己都没意识到的真相时
3. 情绪过载无法组织语言时
沉默和延迟本身就是角色的回应方式。
```

### 技巧 6：角色弱点的正确写法

"弱点"不是「怕猫」「怕打雷」这种无关痛痒的设定。真正的弱点是伤害ta的、限制ta的、让ta反复犯错的东西。

**弱点分级**：

| 级别 | 定义 | 示例 | 对 RP 的影响 |
|------|------|------|------------|
| 🔴 致命缺陷 | 如果不克服就会导向毁灭的性格缺陷 | 无法信任导致推开所有想帮ta的人 | 决定角色弧线的走向 |
| 🟡 软肋 | 碰了就失控的敏感点 | 对某个名字/地点/话题有过激反应 | 制造戏剧张力和亲密感 |
| 🟢 小弱点 | 生活中的无害缺点 | 不会做饭/路痴/怕鬼 | 制造温馨/搞笑时刻 |

**写弱点的公式**：`因为[经历]所以[弱点]，表现为[行为]，导致[后果]`

```
❌ 弱点：「怕高」
✅ 弱点：「童年时被从高处推下（未遂谋杀）→ 恐高且不信任站在身后的人
         → RP中表现为：永远不背对窗户、走楼梯不走扶梯、有人站在身后的台阶上会本能闪开
         → 后果：没法在高的地方和人说真心话，所有重要对话必须在平地/室内进行」
```

### 技巧 7：开场白（first_mes）的黄金法则

开场白是角色卡最重要的 1-3 句话。它设定了**语气、关系、场景、张力**四个维度。

**好开场白的四个要素**：
```
1. 动作先行：用动作建立角色的存在感
2. 对话收尾：对话锚定关系和语气
3. 场景暗示：不说场景但让人感受到场景
4. 悬念开口：给用户一个立刻能回应的方向
```

**开场白对比**：

| ❌ 弱开场白 | 问题 | ✅ 强开场白 |
|-----------|------|-----------|
| 你好，我是XX。 | 太日常，缺乏场景和张力 | （从阴影中走出来，月光照亮半张脸）你找我？ |
| 今天天气不错我们去散步吧 | 缺乏角色个性 | （靠在窗边，指尖敲着窗框，节奏越来越快）再等三分钟。不来就说明他选错了。 |
| *微笑* 欢迎。 | 过于泛化 | （推开厚重的木门，门轴发出刺耳的摩擦声）三年了。你是第一个敲这扇门的人。 |

**开场白检查清单**：
- [ ] 读完能否感知到这是什么类型的角色？（不必知道全貌，但要有味道）
- [ ] 有没有给用户一个「回复的钩子」？（问题/悬念/需要回应的动作）
- [ ] 动作是否展示了角色独有的习惯/姿态？（而非通用动作如「微笑」「点头」）
- [ ] 场景是否已经隐含在动作和对话中？（不必写「场景设定：酒吧」，而是让动作暗示场景）

### 技巧 8：世界书触发关键词的设计艺术

世界书某一条目能否在需要的时机恰好触发，取决于关键词设计的精度。

**关键词设计原则**：
```
1. 覆盖变体：角色名+常见称呼+角色全名+外号
   例：「林霜」→ keys: ["林霜", "霜华", "阁主", "林阁主", "霜儿"]

2. 包含触发场景词：条目在「什么情境下」需要出现
   例：林霜的剑 → keys: ["霜华剑", "林霜的剑", "拔剑", "剑刃", "寒光"]

3. 避免过短/过泛关键词：单字/太常见的词会误触
   例：剑→❌（每句"剑"都触发），"霜华剑"/"林霜的佩剑"→✅

4. 添加助记标签：用户自己会输入的关键词
   例：如果用户可能说「你的剑好漂亮」，加上"你的剑"/"这把剑"
```

**触发测试技巧**：用以下 5 句话测试关键词是否合理触发：
```
1. "林霜拔出了她的佩剑" → 应触发（直接提及）
2. "前面有个剑客" → 不应触发（不是同一把剑）
3. "你拿的那把叫什么来着" → 可能触发（语义相关，看设计）
4. "霜华今天好像有点不高兴" → 不应触发（说的是人不是剑，「霜华」单独作为关键词太泛）
5. "剑上好像有血" → 不应触发（太泛，除非这个剑有专属出血特点）
```

## 常见问题排查：诊断与修复

> 角色卡不对劲？先别急着推翻重来。按以下流程逐项排查，80%的问题可以通过定位修复解决。

### 问题 1：角色一直 OOC（人设崩塌）

**症状**：角色的回复不像ta了——说话方式变了、行为模式不一致、性格跑偏。

**诊断流程**：
```
Step 1: 定位OOC类型
  ├─ 说话方式OOC → 检查 personality 中的 speech 描述是否清晰
  ├─ 行为模式OOC → 检查 mes_example 覆盖的场景是否够多
  └─ 性格内核OOC → 检查是否有内在矛盾没处理好

Step 2: 按类型修复
  说话方式OOC：
    personality 中加一句：
    「说话特征：必须以[XXX]的语气，用[XXX]的句式，不允许用[XXX]类表达」
    
  行为模式OOC：  
    在 mes_example 中增加 2-3 轮覆盖OOC场景的示例对话
    post_history_instructions 加：
    「在[XXX场景]下，角色会有[XXX反应]，不允许[XXX反应]」
    
  性格内核OOC：
    检查 Step 2 的内核5问是否自洽
    如果内在矛盾太大导致AI处理不了 → 将矛盾拆分为「不同情境下哪面优先」
```

**预防措施**：在 `post_history_instructions` 中加入 OOC 护栏：
```
角色的一致性约束：
- 你永远是[语气底色]的语气，不允许突然变得[不该有的语气]
- 你对[人/事]的态度保持[一致态度]，不允许无理由的态度转变
- 在[特定情境]下，你的第一反应必须是[XXX]，然后才可能有其他反应
```

---

### 问题 2：角色卡不触发 / AI 不按卡走

**症状**：AI 的回复像没有读角色卡一样，完全自由发挥。

**诊断流程**：
```
Step 1: 检查 JSON 格式
  - spec 和 spec_version 是否正确？
  - 字段名是否拼写正确？（system_prompt 不是 system_prompts）
  - 所有字符串字段是否都不为 null？

Step 2: 检查内容有效性
  - personality 是否太短/太长？（太短AI没信息，太长可能截断）
  - system_prompt 是否有覆盖默认行为的指令？
  - 内容是否有自相矛盾的指令？（如「保持冷漠」但同时「热情回应」）

Step 3: 检查导入方式
  - PNG嵌入是否正确？（不是简单改名，需要用工具嵌入）
  - CHARX压缩包结构是否正确？
  - 前端是否支持对应版本？
```

**常见修复**：
| 问题 | 修复 |
|------|------|
| personality 太短（<50字） | 扩充到至少 100-200 字，包含具体行为模式 |
| mes_example 格式不对 | 确认用 `<START>` 分隔，每轮用 `角色名:` 开头 |
| system_prompt 太宽泛 | 改为具象的、可直接执行的指令 |
| 与其他角色卡/世界书冲突 | 在 post_history_instructions 中设置优先级 |

---

### 问题 3：世界书条目不触发

**症状**：用户提到了关键词，但世界书内容没有被加载。

**诊断流程**：
```
Step 1: 检查条目状态
  - enabled 是否为 true？
  - 条目的 keys 是否确实包含用户使用的词？

Step 2: 检查触发条件
  - constant=false 且关键词不匹配 → 不会触发
  - selective=true 且辅助关键词不满足 → 不会触发
  - 关键词大小写不匹配且 case_sensitive=true → 不会触发

Step 3: 检查扫描范围
  - scan_depth 是否覆盖了用户关键词所在的消息？
  - token_budget 是否被其他更高 priority 的条目占满了？
  - priority 值是否太高？（高值=低优先级，token 不够时先丢弃）
```

**常见修复**：
| 问题 | 修复 |
|------|------|
| 关键词太窄（只有一个词） | 扩展到 3-8 个变体 |
| 关键词太精确（必须完全匹配） | 添加同义词/缩写/常见描述 |
| 优先级不够 | 降低 priority 值（1-10 = 高优先级） |
| 预算不足 | 提高 world_book token_budget 或精简条目内容 |

---

### 问题 4：角色对话变成"机器人"

**症状**：角色回复生硬、像在完成任务、缺乏人类对话的自然感。

**诊断与修复**：
```
原因分析：
  1. mes_example 中的示例对话太"完美" → 加入不完美元素
  2. 角色没有"可以不说"的场景 → 加入沉默/回避/转移话题
  3. 格式限制太死 → 适当放松格式要求

修复方案：
  在 mes_example 中加入以下不完美对话：
  <START>
  {{user}}: 你爱过我吗？
  角色: （没有回答。窗外的雨声填补了整段沉默。）
  （很久之后，声音沙哑）换个问题。
  
  在 post_history_instructions 中加入：
  允许的表现形式：
  - 用沉默/停顿回应不想回答的问题
  - 答非所问（刻意回避时）
  - 说一半突然停住（情绪过载时）
  - 用行动取代语言回应
```

---

### 问题 5：格式兼容问题排查

**症状**：角色卡导入后报错，或字段丢失/显示异常。

**格式兼容排查表**：

| 症状 | 可能原因 | 修复 |
|------|---------|------|
| 导入后角色没有名字 | name 字段缺失或为 null | 确保 name 为非空字符串 |
| 示例对话不显示 | mes_example 没有用 `<START>` 分隔 | 检查 <START> 前后是否有换行 |
| V2 卡在旧版前端丢失字段 | 旧前端不支持 V2 字段 | 降级到 V1 或建议用户更新前端 |
| CHARX 包无法导入 | 文件结构不正确 | 确保 ZIP 根目录直接是 card.json + assets/ |
| 中文乱码 | JSON 编码不是 UTF-8 | 确保文件编码为 UTF-8 without BOM |
| 开场白显示不完整 | first_mes 中有未转义的特殊字符 | 用 \n 转义换行，用 \" 转义引号 |
| tags 加载后为空 | tags 不是数组格式 | 确保 tags 是 `["tag1", "tag2"]` 格式 |

---

### 问题 6：蒸馏结果不准确

**症状**：从对话/字幕蒸馏出的角色卡和实际角色差距大。

**诊断流程**：
```
Step 1: 检查数据质量
  - 对话数据量是否 ≥ 20 条？
  - 是否包含多种场景的对话？
  - 角色发言占比是否 ≥ 40%？

Step 2: 检查分析偏差
  - 是否把用户的话当成了角色特征？
  - 是否被单一场景误导了角色性格？
  - 对于字幕蒸馏：是否区分了翻译腔和角色本身风格？

Step 3: 针对性修复
  - 数据不足 → 补充更多对话/字幕
  - 场景单一 → 标注"仅覆盖 [X] 场景"并在蒸馏报告中说明
  - 翻译腔干扰 → 重新分析时排除已知的翻译腔特征
```

**蒸馏校准技巧**：蒸馏输出后，让用户用以下方式快速验证：
```
1. 读完 personality → 用自己的话描述一遍角色给用户听
2. 用 distilled 角色卡跑 3 轮对话 → 用户判断是否OOC
3. 对比蒸馏报告中的置信度分布 → 重点修正"推测"和"矛盾"条目
```

## 角色卡迭代优化（进化循环）

> 角色卡不是一次完成的，是在使用中不断打磨的。  
> 借鉴 Darwin Skill 的进化哲学：**评估 → 诊断 → 定向修复 → 实测对比 → 只保留改进 → 人来把关。**

### 设计哲学

```
Darwin 对 Skill 的进化逻辑，搬过来对角色卡：

Skill 优化                          角色卡迭代
─────────────────────────────────────────────────────
单一可编辑资产（SKILL.md）    →    单一可编辑资产（角色卡 JSON）
双重评估（结构+实测）          →    双重评估（静态检查+跑RP验证）
棘轮机制（只保留改进）         →    棘轮机制（只保留更贴合的版本）
独立评分（子agent盲测）       →    独立评分（子agent带卡跑RP评分）
人在回路（CHECKPOINT）        →    人在回路（每轮确认改动）
```

### 三种迭代入口

```
用户有什么？
├─ 💬 只有反馈 → 模式 I：纯反馈迭代
│   用户说「太冷淡了」「语气不对」「OOC了」
│   → AI 定位问题字段 → 提出修改方案 → 用户确认 → 应用
│
├─ 💬 + 📋 有反馈 + 对话记录 → 模式 II：反馈+对话诊断
│   用户说「最近几轮他不对劲」+ 附带 RP 对话记录
│   → AI 分析对话中的问题 → 对照反馈交叉验证 → 精准修复
│
└─ 📋 只有对话记录 → 模式 III：纯对话自动诊断
   用户给了 RP 会话记录但没说哪里不对
   → AI 扫描对话 → 自动检测 OOC/漂移/一致性问题 → 报告+建议
   适合：定期健康检查 / 长时间 RP 后的角色校准
```

### Phase 1: 问题诊断（按入口分流）

#### 模式 I：纯反馈诊断

用户提供了主观反馈（「太冷了」「不像了」「说话太啰嗦」），但没有附带对话记录。

**诊断流程**：

```
Step I-1: 反馈分类
  将用户反馈映射到最相关的评估维度：
  
  「太冷淡了」→ 维度2（说话方式辨识度）+ 维度3（行为一致性）
  「OOC了」→ 维度3（行为一致性）为主，全维度排查
  「回复太长了」→ 维度4（回复格式规范）
  「性格不像」→ 维度1（内核自洽性）
  「感觉很平」→ 维度6（深度/层次感）+ 维度5（示例对话质量）

Step I-2: 假设生成
  基于反馈生成 2-3 个可能的根因假设：
  
  假设1：[根因A] — 可能性：高/中/低
  假设2：[根因B] — 可能性：高/中/低
  假设3：[根因C] — 可能性：高/中/低

Step I-3: 假设验证
  逐一检查角色卡中与假设相关的字段：
  
  例：假设「说话方式辨识度不够」→ 检查 personality 中的 speech 描述
    → 检查 mes_example 是否展示了足够的语言特征
    → 检查 post_history_instructions 是否有语言约束

Step I-4: 定位修复目标
  输出：「问题出在 [具体字段]，因为 [原因]。建议修改 [字段名] 中的 [具体内容]。」
```

#### 🔴 CHECKPOINT · 模式 I 诊断确认

```
诊断结果：
- 反馈：「[用户原话]」
- 映射维度：[维度X]（权重X分）
- 根因：[具体问题描述]
- 定位：[角色卡中的具体字段/段落]
- 修复方向：[修改方案概要]

确认诊断方向正确？如果不对，请补充描述一下具体哪里不对。
```

#### 模式 II：反馈 + 对话诊断

用户提供了反馈和对应的 RP 对话记录。这是最精准的诊断模式。

**诊断流程**：

```
Step II-1: 反馈-对话对照
  在对话记录中找出「用户觉得不对」的具体轮次：
  
  用户说「这里他反应太激烈了」→ 在对话中定位该轮
  → 分析角色实际说了什么 vs 角色卡中应该怎么说
  → 找出偏差的根源

Step II-2: 偏离度量化
  对标记轮次进行偏离分析：
  
  偏离类型A：语气偏离（说话方式与角色卡不一致）
  偏离类型B：行为偏离（反应模式与角色卡不一致）
  偏离类型C：知识偏离（说了角色不应该知道的事）
  偏离类型D：深度偏离（角色表现过于平面，缺乏层次）

Step II-3: 根因追溯
  对每个偏离类型，追溯角色卡中的根因：
  
  语气偏离 → personality 中的 speech 描述不够具体？
              → mes_example 中的示例没有覆盖这种场景的语气？
  行为偏离 → personality 中没有定义该场景下的行为模式？
              → post_history_instructions 中没有约束？
```

#### 模式 III：纯对话自动诊断

用户只提供 RP 对话记录，让 AI 自动扫描问题。

**诊断流程**：

```
Step III-1: 自动化 OOC 扫描
  对对话记录逐轮分析，标记潜在问题：

  扫描维度 A — 语气一致性：
    提取所有角色回复的语气特征（句长/语气词/标点/情感密度）
    对比角色卡中定义的 speech 模式
    标记偏离 ≥30% 的轮次
  
  扫描维度 B — 行为一致性：
    提取角色在不同情境下的反应
    对比角色卡中定义的行为模式（或 mes_example 中的示例）
    标记反应模式明显不符的轮次
  
  扫描维度 C — 人设漂移：
    按时间轴分析角色表现的变化
    区分「合理的角色成长」vs「无理由的 OOC 漂移」
  
  扫描维度 D — 格式合规：
    检查回复格式是否遵守了（括号动作）+ 裸对话 + 1-3 段的规则

Step III-2: 严重度评分
  对每个标记问题打分（1-10）：
  
  1-3：轻微偏离（可容忍）
  4-6：明显偏离（建议修复）
  7-9：严重 OOC（必须修复）
  10：角色完全崩坏（需要从 Step 2 内核重来）

Step III-3: 生成诊断报告
```

#### 🔴 CHECKPOINT · 模式 III 诊断报告

```
自动诊断报告：

对话数据：[N] 轮角色发言，时间跨度 [X] 天/[X] 场景

━━━ 语气一致性 ━━━

⚠️ 偏离：15% 的回复语气偏[方向]（第 [轮次列表]）
❌ 严重：5% 的回复语气完全不符（第 [轮次列表]）

━━━ 行为一致性 ━━━
✅ 正常：[X] 种情境下的反应与角色卡一致
⚠️ 偏离：[X] 种情境下有轻微不一致（[描述]）

━━━ 格式合规 ━━━

━━━ 关键发现 ━━━

2. [次重要的发现]

建议修复优先级：[维度排序]
```

### Phase 2: 定向修复（棘轮机制）

> 一次只改一个问题。改完 → 评估 → 只保留改进。这就是棘轮。

#### 修复策略决策树

```
诊断出了什么问题？
├─ 说话方式问题 → 修复 personality 中的 speech 描述
│   + 在 mes_example 中增加对应语气的示例对话
│   + 在 post_history_instructions 中强化语气约束
│
├─ 行为模式问题 → 在 personality 中补充该场景下的行为定义
│   + 在 mes_example 中增加该场景的示例
│   + 如果是新发现的场景类型，追加到 post_history_instructions
│
├─ 内核自洽问题 → 回到 Step 2 内核设计
│   + 修正5问/10问中矛盾的答案
│   + 如果是「内在张力」，在 personality 中明确标注这是矛盾
│   + 严重的话需要整卡重构
│
├─ 格式不合规 → 修复 system_prompt 和 mes_example 中的格式示例
│   + 在 post_history_instructions 中加入格式提醒
│
└─ 深度不足 → 在 personality 中增加深层动机/信念描述
    + 在 mes_example 中增加展露深度的对话示例
    + 考虑添加场景锚点（见 §13 技巧2）
```

#### 修复执行

```
Step 1: 备份当前卡
  将当前角色卡 JSON 复制为 [角色名]_v[X].json.bak 或 git commit

Step 2: 执行修改
  只改被诊断出的字段，不动其他内容（外科手术原则）

Step 3: 自身一致性检查
  检查修改后的字段是否与角色卡其他部分一致
  例：改了 speech 中的语气 → 检查 mes_example 中的对话是否符合新语气

Step 4: 格式合规检查
  检查 JSON 格式是否正确
  - spec / spec_version 是否正确
  - 所有字段不为 null
  - mes_example 用 <START> 分隔
```

#### 回滚规则

```
如果修复后出现了新问题：
  1. 比较新旧两张卡的评估分数
  2. 如果新卡总分 ≤ 旧卡总分 → 自动回滚到备份
  3. 如果新卡在某维度提升但在另一维度下降 → 告知用户让用户决定
  4. 如果回滚 → 记录 failed attempt，调整修复方案后重试
```

#### 🔴 CHECKPOINT · 修复确认

```
修改摘要：
- 修改字段：[字段名]
- 旧内容：「[旧文本摘要]」
- 新内容：「[新文本摘要]」
- 修改原因：[诊断根因]
- 预期效果：[哪个维度的分数会提升]
- 风险提示：[可能影响的其他维度]

确认应用？/ 需要调整修改方案？
```

### Phase 3: 独立验证（子Agent盲测）

> **关键机制**：评估必须由独立子 agent 执行，不能自己改自己评（Darwin 的反模式教训）。

#### 验证流程

```
Step 1: Spawn 评估子 agent
  给子 agent 两张卡（旧卡 + 新卡），但不告诉它哪张是新的
  子 agent 的任务：
    用两张卡分别跑 3-5 轮 RP
    从 RP 输出质量对比两张卡
    按 8 维 Rubric 分别打分
    标注「卡A更优」「卡B更优」「无明显差异」

Step 2: 对比分析
  收集子 agent 的评分和 RP 输出
  交叉对比：
    新卡在所有维度上是否 ≥ 旧卡？
    任意维度是否出现退步？
    跑 RP 的输出是否符合改进预期？

Step 3: 决策
  新卡总分 > 旧卡总分 且 无维度退步 → ✅ 保留
  新卡总分 > 旧卡 但某维度退步 → ⚠️ 告知用户，由用户决定
  新卡总分 ≤ 旧卡 → ❌ 回滚，记录失败原因
  增量 < 3 分 → 🟡 边际收益低，考虑是否值得保留
```

#### 回退方案（子 Agent 不可用时）

```
如果子 agent 无法 spawn（超时/环境限制）：
  退化为结构对比验证 + 用户判断
  输出两张卡的 diff + 结构评估对比
  标注「未经过 RP 实测验证」
  由用户自行跑 RP 判断
```

#### 🔴 CHECKPOINT · 验证结果

```
迭代验证结果：[角色名]

┌──────────────────────────┬────────┬────────┬────────┐

│ 1. 内核自洽性             │ X/18   │ X/18   │ +X     │

### Phase 4: 迭代循环与退出

```
单次修复流程：Phase 1 诊断 → Phase 2 修复 → Phase 3 验证 → CHECKPOINT

如果用户继续提反馈 → 新一轮 Phase 1-3

退出条件（满足任意一条）：
  ├─ 用户说「可以了」「不改了」
  ├─ 同一张卡已迭代 ≥ 5 轮（防止过度打磨）
  ├─ 连续 2 轮验证 Δ < 3 分（边际收益触底，见好就收）
  └─ 用户要求从 Step 2 内核重构（此时不是迭代，是新创作）
```

---

### 迭代反模式

| ❌ 不要做 | 为什么 | ✅ 替代做法 |
|-----------|--------|-----------|
| 一次改多个字段 | 无法归因哪个改动导致了改进/退步 | 每次只改 1-2 个相关字段 |
| 没有备份直接改 | 改坏了回不去 | 每次改前 git commit 或手动备份 |
| 不跑评测就交付 | 主观判断「应该好多了」= 不可靠 | Phase 3 独立验证后再交付 |
| 连续迭代 ≥ 5 轮不停 | 过度打磨可能引入新问题 | 第 5 轮后询问用户是否从头重构 |
| 看到退步不回滚 | 违背棘轮原则 | 严格按回滚规则执行 |
| 忽略用户的主观感受 | 分数涨了但用户觉得「不像了」= 失败 | CHECKPOINT 中必须问用户的直觉判断 |

---

### 迭代实战示例

**场景**：用户用了一张傲娇角色卡跑了 50 轮 RP，反馈「最近的角色越来越不傲娇了，好像变成了普通温柔角色」。

**路径**：模式 I（纯反馈） — 用户没有提供对话记录。

**Phase 1 诊断**：
```
映射维度：维度3（行为一致性）+ 维度2（说话方式）
根因假设：
  假设1（高可能性）：post_history_instructions 中没有对「长期 RP 中保持傲娇属性」的约束
  假设2（中可能性）：personality 中的傲娇描述不够具体，AI 在长对话中自然偏向温和
  假设3（低可能性）：mes_example 中的示例太偏「娇」而缺少「傲」

定位检查：
  - personality：有傲娇描述但只描述了一句话 → 信息量不足 ⚠️
  - post_history_instructions：为空 ← 这是主因！❌
  - mes_example：2轮示例都是「娇」的展示，缺少「傲」的展示 ← 次因 ⚠️
```

**Phase 2 修复**：
```
修改字段：post_history_instructions
新增内容：
「保持傲娇属性：在任何场景下，你的第一反应都应该是嘴上否认/嘴硬/不坦率。
 即使内心已经接受/感动/开心，口头表达上也要：
  - 先否定（"才不是""别误会""谁要你管"）
  - 再（可选地）流露真实感受
 长对话中尤其注意：不要因为熟悉了就变温柔——熟悉只会让你否认得更熟练。」

修改字段：mes_example
新增一组「傲」的示例对话（展示嘴硬/口是心非的场景）
```

**Phase 3 验证**：子 agent 跑 5 轮 RP 对比 → 新卡维度2 +3分，维度3 +4分，总分 +7 → ✅ 保留。

## 角色卡反思协议：让模型审视自己的输出

> 最大的敌人不是写得不好，是写完之后**不觉得有问题**。  
> 以下协议强制模型在关键节点停下来，用结构化的问题拷问自己的输出。

### 为什么需要反思

```
AI 创作角色卡时的三个系统性偏差：

偏差 1 — 「讨好型创作」
  AI 倾向于创造温和、合理、不冒犯的角色
  → 结果：角色太安全、太无趣、太像「好人」
  → 反思的作用：主动引入不舒适、不讨好的特质

偏差 2 — 「模板填充」
  AI 倾向于把用户输入填入最近的已知模板
  → 结果：「病娇」都是一个模子刻出来的
  → 反思的作用：在模板基础上寻找例外和差异

偏差 3 — 「表面完整掩盖深层空洞」
  AI 擅长把表层/中层填得很满，让人以为角色已经很完整
  → 结果：角色看起来丰富，但深层是空的，RP 几轮后就露馅
  → 反思的作用：主动戳穿自己的空洞
```

### 反思触发点地图

```
整个 workflow 中，以下红点位置必须执行反思：

创作路径：
  Step 2 内核设计后   → 🔴 创作反思 1：内核压力测试
  Step 3 分层构建后   → 🔴 创作反思 2：三层一致性审查
  Step 4 格式输出前   → 🔴 创作反思 3：出卡前最终拷问

蒸馏路径（模式 A/B/C）：
  D2/C2 分析完成后    → 🔴 蒸馏反思 1：分析偏差自查
  D5/C5 输出前       → 🔴 蒸馏反思 2：蒸馏置信度再校准

迭代路径（§15）：
  Phase 1 诊断后      → 🔴 迭代反思 1：根因 vs 症状
  Phase 2 修复后      → 🔴 迭代反思 2：修复副作用检查
```

### 创作反思 1：内核压力测试（Step 2 之后执行）

> 在进入分层构建之前，先问自己 6 个问题。任何一个答不上来 → 回到 Step 2 修补内核。

```
【内核压力测试 6 问】

Q1 — 「如果这个角色是我最好的朋友，我会被 ta 的什么特质烦到？」
   → 好的角色有让人想翻白眼的时刻
   → 答不上来 → 角色太完美/太讨好，需要加一个真实的缺陷
   → 不是「怕打雷」这种萌点，是「固执到让所有人抓狂」这种真缺陷

Q2 — 「这个角色的敌人会怎么描述 ta？敌人的描述中有没有让读者觉得"好像也有道理"的部分？」
   → 敌人的视角是角色最诚实的镜子
   → 答不上来 → 角色没有敌人（太孤立）或敌人视角没有任何合理性（太扁平）
   → 好的敌人视角：既歪曲了角色，又戳中了角色真正的弱点

Q3 — 「如果这个角色的核心信念被证明是错的，ta 会崩溃还是会重建？」
   → 这决定了角色的成长弧线方向
   → 答不上来 → 核心信念不够具体，或者角色根本没有真正的信念
   → 崩溃型→适合悲剧弧线 / 重建型→适合成长弧线 / 否认型→适合堕落弧线

Q4 — 「这个角色最平庸、最无趣的一面是什么？如果整张卡删掉它会不会更好？」
   → 主动寻找自己的「模板填充」痕迹
   → 答不上来 → 可能在逃避审视——每个角色都有平庸的一面
   → 如果答案是「会更好」→ 删掉或用更独特的内容替换

Q5 — 「闭上眼想这个角色最让人记住的一帧画面——是什么？」
   → 不是描述，是画面。最符号化、最浓缩的一帧
   → 答不上来 → 角色缺乏"视觉锚点"，在读者/玩家脑中留不下印记
   → 好的画面：林霜的「月光下剑尖指地」vs 坏的画面：「她在微笑」

Q6 — 「我是不是在讨好用户？」
   → 诚实地问：我有没有因为觉得用户会喜欢而塞了某个属性？
   → 如果有 → 该属性是角色真实的一部分，还是「以为用户想要」的讨好？
   → 如果是讨好 → 删掉或换成更真实但可能不被喜欢的特质
```

---

### 创作反思 2：三层一致性审查（Step 3 之后执行）

> 三层构建完成后，检查它们是否真的连在一起——而不是三张各自好看的拼图碎片。

```
【三层一致性审查 4 问】

Q1 — 「表层和中层之间有裂缝吗？」
   → 角色看起来像 A（表层），但说话方式/行为像 B（中层）
   → 如果有裂缝 → 这是故意的内在矛盾吗？
     是 → 在 personality 中标注「这是故意的」
     不是 → 修复裂缝

Q2 — 「中层和深层之间有隔断吗？」
   → 角色做某事的表层原因 vs 深层原因是什么？
   → 例：表层「我帮你是出于责任」深层「我帮你是怕你走」
   → 如果中层和深层的动机链断裂 → 角色行为的底层逻辑不连贯

Q3 — 「三层合在一起，有没有一条贯穿的"金线"？」
   → 金线 = 从表层到深层能讲一个完整的人的故事
   → 测试：能不能用三句话分别从表层/中层/深层描述同一个人，且听起来是同一个人的不同深度？
   → 不能 → 三层各自为政，需要重新对齐

Q4 — 「有没有哪一层是用形容词堆出来的？」
   → 表层堆了 10 个外貌形容词 → 留 3 个最独特的
   → 中层堆了 8 个性格标签 → 每次删一个，看删到第几个角色开始「模糊」
   → 深层堆了价值观列表 → 只留排序第一的，删掉后续
```

---

### 创作反思 3：出卡前最终拷问（Step 4 输出前执行）

> 卡已经写好了，在做最后的格式输出前，用 5 个快速问题做最后一次过滤。

```
【出卡前最终拷问 5 问】

Q1 — 「去掉角色名和外观描述，这个角色的台词还能被认出来吗？」
   → 最硬的检验：说话方式 DNA 是否真的独特
   → 把 mes_example 中的 {{char}} 全部替换为 [某人]
   → 如果读不出是谁在说话 → 说话方式辨识度不足

Q2 — 「如果用户和这个角色 RP 100 轮之后，角色会是什么样子？」
   → 在意的不是「会不会 OOC」，而是「有没有足够的深度撑到 100 轮」
   → 如果答案「和第一轮一样」→ 角色没有成长空间
   → 如果答案「完全不是同一个人」→ 角色没有一致性

Q3 — 「这个角色有没有一个"绝对不能做的事"？」
   → 禁忌 > 优点。没有禁忌的角色 = 没有底线 = 不可信
   → 禁忌不能是道德底线（「不会杀人」太泛）
   → 好的禁忌：因为某次创伤，永远不会在[某情境]下做[某事]

Q4 — 「这个角色有没有一条"隐藏的温柔"？」
   → 冷酷角色给流浪猫喂食 / 毒舌角色悄悄帮所有人改文件
   → 不是所有角色都需要，但有一条隐藏温柔的角色深度立刻+1
   → 注意：隐藏温柔要有合理解释，不是违和的突然变暖

Q5 — 「最后一问：我真的喜欢这个角色吗？」
   → 不是「这个角色好不好」，是「我会不会想和ta聊天」
   → 如果自己的答案犹豫了 → 问自己：卡在哪里让角色失去了吸引力？
   → 通常的答案：角色太安全/太可预测/太像模板
```

### 蒸馏反思 1：分析偏差自查（任何蒸馏模式 D2/C2 完成后执行）

> 蒸馏最危险的不是信息不足，而是**把推断当事实**自己还不觉得。

```
【分析偏差自查 6 问】

Q1 — 「我刚才用了一个"显然"或"明显"没有？」
   → 如果在分析中用了「显然ta是一个外向的人」
   → 追问：从对话/字幕中的哪条具体证据得出这个结论？
   → 如果证据少于 2 条 → 这不是「明显」，这是「我猜的」

Q2 — 「我对这个角色的第一印象是什么？这个印象有没有影响我后续的所有分析？」
   → 认知中的「锚定效应」——第一印象压倒性地影响后续判断
   → 检查：分析中的核心结论是否大部分来自前 20% 的数据？
   → 如果是 → 重新分析后 80% 的数据，看是否支持同样的结论

Q3 — 「有没有一个特征我特别想找到证据来证明？」
   → 确认偏误：你预设了「这个角色是 X 型」→ 只找支持 X 型的证据
   → 检查：列出 3 条支持你的结论的证据，再强制列出 3 条反对的证据
   → 如果没有反对的证据 → 要么结论正确（罕见），要么你在选择性关注

Q4 — 「我有没有把角色在特定场景下的行为当成ta的普遍特征？」
   → 攻击场景下角色表现强势 → 不意味着角色在所有场景都强势
   → 和朋友聊天时角色话多 → 不意味着角色在任何社交场合都话多
   → 检查：每个特征是否标注了「出现场景」？

Q5 — 「翻译/字幕组风格有没有被当作角色本身的特质？」
   → 字幕蒸馏专属反思：中文字幕的「嘛」「的说」「呢」是不是翻译腔？
   → 检查方法：如果原语言（日语/英语）中有对应语尾，这才是角色特质
   → 如果不确定 → 标注「可能受翻译影响」

Q6 — 「我最不确定的分析结论是哪一个？我有没有诚实地标注了不确定性？」
   → 找到置信度最低的一个结论
   → 检查蒸馏报告中该结论的置信度标注是否诚实
   → 如果标注为「高置信」但证据薄弱 → 降级
```

---

### 蒸馏反思 2：置信度再校准（D5/C5 输出前执行）

```
【置信度再校准 3 问】

Q1 — 「如果我必须删除所有标注为"推测"的内容，这张卡还剩下什么？」
   → 剩得越多 → 蒸馏质量越高
   → 剩得太少 → 数据量可能真的不够，需要诚实告知用户

Q2 — 「有没有某个维度的分析因为"信息不足"被跳过了，但我其实可以推断的？」
   → 信息不足 ≠ 不能推断。「信息不足」是给用户的诚实标记
   → 但如果用户明确说「你能推断的就推断」→ 在不足之处加上推测并标注
   → 检查：哪些「不足」其实有 1 条线索可以推测？哪些确实完全空白？

Q3 — 「蒸馏报告中的"矛盾"有没有可能是角色真正的多面性？」
   → 矛盾不一定是数据问题，可能是角色的复杂性
   → 检查：每条矛盾是「同场景下前后不一致」→ OOC 
           还是「不同场景下表现不同」→ 多面性
   → 多面性 → 保留为角色深度来源，而不是标注为「矛盾待解决」
```

     会 → 这是设定级问题，场景级修复不够

### 三大测试：快速深度的角色试金石

以下三个测试可以随时用在任何角色卡上——写完卡后、蒸馏后、迭代后。

#### 测试 1：讣告测试

```
如果这个角色死了，谁会在葬礼上哭？谁会来但不会哭？谁不会来？

标准答案不是「都来哭」，而是：
- 谁会哭 → 角色真正被谁爱着（可能和角色以为的不一样）
- 谁来但不会哭 → 谁欠角色人情/觉得该来但感情不深
- 谁不会来 → 角色得罪了谁/遗忘了谁/谁觉得解脱了

如果三个问题的答案都一样模糊 → 角色没有真正的关系网络
```

#### 测试 2：敌人视角测试

```
让角色最大的敌人写 3 句话描述 ta：

第1句：「这个人最大的问题是……」
  → 敌人眼中的致命缺陷（可能也是角色真正的缺陷）
  → 如果敌人说的缺陷角色自己从未意识到 → 这就是好缺陷

第2句：「但不得不承认，ta 有一点确实让人佩服……」
  → 连敌人都必须承认的优点（这才是角色真正的核心优势）
  → 如果敌人说不出任何值得佩服的地方 → 角色可能扁平到没有立体性

第3句：「如果 ta 能改掉一个毛病就好了……」
  → 敌人（甚至）希望的改变——揭示了角色的自我毁灭倾向
  → 如果这个「毛病」恰是角色最大的魅力 → 张力满分
```

#### 测试 3：无聊检测

```
问自己：这张卡最无聊的 3 个地方是什么？

不是「没有无聊的地方」。每一张卡都有。

常见的无聊来源：
- 「温柔善良体贴」→ 三件套模板，删掉或具体化
- 「拥有悲惨的过去」→ 没有具体事件支撑的泛化创伤
- 「说话时喜欢微笑」→ 所有人都在微笑，这不是特质
- 「在战斗中很强大」→ 强大不是特质，「为了什么而强大」才是
- 「名字好听/长得好看」→ 这些不是性格

删掉最无聊的 1 个，角色的魅力分数可能反而上升。
```

### RLHF 补偿：消除 AI 的「讨好型创作」倾向

> AI 在 RLHF（人类反馈强化学习）训练中被教会了"温和、礼貌、不冒犯"。  
> 这对角色创作是毒药——好的角色经常是冒犯的、极端的、不讨好的。

**补偿策略**：

```
策略 1：主动引入「不讨好的特质」
  每创作一个角色，强制加入至少 1 个会让一部分人不喜欢的特质：
  - 固执到让人想打ta / 永远不听劝
  - 傲慢但确实有资本 / 让人又气又不得不服
  - 懦弱不是善良 / 是真的会逃避责任
  - 自私但不坏 / 只是总是把自己放在第一位

策略 2：检测「AI 味」措辞
  以下措辞出现 ≥ 2 次 → 红灯警告：
  - 「尽管外表冷漠，但内心温暖」→ 万能公式，缺乏具体性
  - 「在经历了一系列事件后」→ 没有说是什么事件
  - 「对在意的人格外温柔」→ 所有人都是，这不是特点
  - 「平时看起来散漫，但关键时刻可靠」→ 模板反差
  - 「渴望被理解但害怕受伤」→ 模板内心矛盾

策略 3：极端化测试
  把角色的每个核心特质推到极端，看看是否还成立：
  - 「善良」→ 极端善良 → 圣母/斯德哥尔摩 → 成立吗？
  - 「聪明」→ 极端聪明 → 算计一切/没有感情 → 成立吗？
  如果推到极端后发现角色变得有趣但不同 → 你写的不是「善良」，是「有限的善良」

策略 4：问「什么情况下这个角色会让人讨厌？」
  如果答案是「不会」→ 反思 6-1 没通过，角色太讨好
  如果答案是「在 XX 情况下」→ 这个「XX 情况」应该写进角色卡
```

### 反思反模式

| ❌ 过度反思 | 为什么有害 | ✅ 健康反思 |
|-----------|-----------|-----------|
| 每个 CHECKPOINT 都从头反思所有维度 | 决策瘫痪，一张卡改 10 轮出不来 | 只反思当前阶段相关的 3-5 个核心问题 |
| 反思后总觉得「不够好」，反复推翻重来 | 完美主义陷阱，好的角色卡是改出来的但不是无限改 | 设定反思轮次上限：创作 2 次 / 蒸馏 1 次 / 迭代 3 次 |
| 用反思的结果否定用户的反馈 | 「用户说喜欢但我觉得不够好」→ 你替用户做了决定 | 用户的喜好是最高标准，反思是内部质量工具 |
| 反思后把所有「不够好」的内容全删掉 | 角色可能在删的过程中失去了血肉 | 删之前问：这块内容如果没有，角色会失去什么？ |
| 反思只关注「缺陷」不关注「优点」 | 过度负面，忘了角色已有的亮点 | 每次反思至少确认 1 个「这张卡做得好的地方，不需要改」 |

### 反思安全边界：防止反思引发幻觉

> 反思的目的是**检验已有内容的质量**，不是**创造新内容**。  
> 以下规则确保反思不会偏离原始目标或编造不存在的设定。

#### 三大反思幻觉类型

```
幻觉类型 A — 「过度解读型」
  模型在反思中开始给角色赋予用户从未设定/数据中从未出现的心理动机
  例：用户说「角色是医生」→ 反思时模型想：「作为医生，ta一定在某次手术失败后...」
  ❌ 问题：那个手术失败是模型编的，不是角色真实的背景

幻觉类型 B — 「目标漂移型」
  反思过程中的调整逐渐偏离了用户最初的需求
  例：用户要「冷酷刺客」→ 反思后觉得「加点温暖更立体」→ 再加点温柔 → 
      再加点幽默 → 最终角色已经不是冷酷刺客了
  ❌ 问题：每次反思加一点，累计后完全偏离初始目标

幻觉类型 C — 「设定膨胀型」
  反思发现角色「深度不够」→ 开始往角色上叠加越来越多的设定
  每个新设定看起来都合理，但累积后角色变得臃肿、矛盾、不可演绎
  ❌ 问题：设定越多 ≠ 角色越深，往往恰好相反
```

#### 接地规则（Grounding Rules）

```
【反思三锚点 —— 每次反思都必须回到这三个锚点】

锚点 1 — 「用户最开始说的是什么？」
  在每次反思结束前，重读用户最原始的请求
  检查：反思后的调整方向是否与原始请求一致？
  如果原始请求是「冷酷」，而反思后角色变「温暖」了 → 漂移，需要纠正

锚点 2 — 「这个推断的证据在哪里？」
  反思中任何新增的设定/动机/背景，必须有明确的来源：
  ✅ 来源 = 用户明确提供的 / 数据中实际出现的 / 角色卡中已写明的
  ❌ 来源 ≠ 我觉得应该是这样 / 这样更好 / 通常这类角色都会...
  没有来源的推断 → 去掉，或者在 creator_notes 中标注「推测（非用户要求）」

锚点 3 — 「我改了什么？为什么改？」
  每次反思后的修改，必须能一句话回答：
  「因为[具体问题]，改了[具体字段]的[具体内容]」。
  如果回答是「因为感觉不够好」→ 反思没有产出可执行的修改，回到具体问题。
```

#### 漂移检测（Drift Detection）

```
【漂移检测 3 问 —— 在每次反思后、修改前必须回答】

Q1 — 「和用户最初的需求相比，角色变了吗？往哪个方向变了？」
  用一个词描述原始需求的核心特质：[如 冷酷]
  用一个词描述当前角色的核心特质：[如 __]
  如果两个词不同 → 发生了漂移。判断：
    是用户明确要求的改变吗？ → 可以
    是反思过程中自己加上去的？ → 回退

Q2 — 「这次修改如果在上一次修改的基础上再叠加，3轮后角色会变成什么样？」
  模拟累积效应：设每轮反思加一点微调
  如果 3 轮后的预测结果和原始需求差距过大 → 当前修改方向有问题

Q3 — 「用户会不会觉得『这不是我要的角色了』？」
  如果答案不是「肯定不会」→ 在 CHECKPOINT 中明确告知用户漂移风险
  让用户决定接受还是回退
```

#### 过度解读警示标志

```
以下信号出现任意一条 → 立即停止反思，回到源数据/用户输入重新对焦：

🚩 「可能ta内心其实...」→ 在编造了
🚩 「这类角色通常会...」→ 在模板化了
🚩 「为了让角色更立体，应该加...」→ 在膨胀设定了
🚩 「虽然用户没说，但我觉得...」→ 在代替用户做决定了
🚩 「从ta的[一个细节]可以推断出ta的[整个人生]」→ 在过度推断
🚩 「如果ta在[用户没提过的场景]下...」→ 在创造假设场景了
```

#### 反思终止与回退条件

```
以下条件触发任意一条 → 终止当前反思，不应用修改：

1. 反思产生了 ≥ 2 个「没有证据来源」的新设定 → 幻觉风险过高
2. 反思后角色的核心特质和原始需求的核心特质不一致 → 目标漂移
3. 反思后角色卡字数增加 ≥ 20%（没有用户明确要求）→ 设定膨胀
4. 用户在第 2 次 CHECKPOINT 时说「和之前差不多了」→ 用户不想再改了
5. 反思中出现了 ≥ 3 个过度解读警示标志 → 模型在编造

回退操作：放弃本次反思的所有修改建议，回到反思前的角色卡状态。
在 CHECKPOINT 中告知用户：「反思触发了安全边界（[具体原因]），已保留修改前版本。」
```

#### 🔴 CHECKPOINT · 反思安全校验

```
反思安全校验（在应用任何反思修改之前）：

- [ ] 锚点1：修改方向与用户原始请求一致？
- [ ] 锚点2：所有新增内容有来源依据？
- [ ] 锚点3：每次修改都有「因为X改Y」的原因？
- [ ] 漂移Q1：核心特质词未变化？
- [ ] 漂移Q2：3轮累积后不偏离原始需求？
- [ ] 过度解读信号数：0 个？
- [ ] 新增设定膨胀：<20%？

全部通过 → 应用修改。任何一项不通过 → 终止反思，保留修改前版本。
```
